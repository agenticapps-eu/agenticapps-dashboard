## Summary

The UI redesign is **a token + shell + primitive migration on top of an unchanged data layer**. The 8 focus areas resolve as follows:

1. **Tailwind 4 token strategy** — split into `packages/spa/src/styles/tokens.css` imported via `@import` from `global.css`. Use the `@theme` directive (NOT `tailwind.config.ts`, NOT `theme.extend`). The existing `global.css` already mixes `@theme` + `:root` blocks; the migration extracts the locked color/type/spacing tokens from UI-SPEC §2-4 into the new `tokens.css` while preserving the `@custom-variant dark` line. **[VERIFIED: Tailwind v4.1 docs at tailwindcss.com/docs/theme — split-file pattern is explicitly recommended.]**

2. **Inter variable font** — install `@fontsource-variable/inter@5.2.8` (single dep, ~1.9MB pre-Vite-tree-shake; only the variable WOFF2 is bundled in production). Import once at `main.tsx` top, before `./styles/global.css`. **[VERIFIED: npm registry 2026-05-09 — version 5.2.8.]**

3. **TanStack Router pathless layout** — use `createRoute({ getParentRoute: () => rootRoute, id: '_appshell', component: AppShellV2 })` with the four paired routes nested under it. `/onboarding` and `/pair` remain direct children of `rootRoute`. The current `rootRoute` already mounts AppShell — flip it to a bare passthrough and move AppShell mounting down to the new pathless layout. **[VERIFIED: TanStack Router v1.169.2 — pathless route via `id` field is the official code-based equivalent of the file-based `_layout` underscore convention.]**

4. **Feature-flag swap** — `import.meta.env.VITE_APPSHELL_V2` read once at module scope in `router.tsx`. Two route-tree shapes built side by side in the same file; Vite tree-shakes the unused branch in prod (env vars are static-replaced at build time). Default OFF in dev → user opts in by setting `VITE_APPSHELL_V2=1` in `.env.local`. Final commit (Wave 5) deletes the flag and the old shell.

5. **Component primitive pattern** — inline className composition with discriminated union variant types. **No `cn()` helper, no `class-variance-authority`, no `clsx` dep.** The codebase already has zero of these (verified: `grep -rn "cn(" packages/spa/src` returns nothing) and the locked discipline of "no third-party JS additions" rules out CVA. Pattern: `function Pill({ variant, ... }: { variant: 'neutral' | 'accent' | ... }) { const cls = VARIANT_CLASS[variant]; return <span className={\`base-classes ${cls}\`} ... /> }` — matches `PanelContainer.tsx` style already in use.

6. **Per-panel migration cost** — **mechanical for 12 of 13 panels**. All 8 Phase 4 panels + 5 Phase 5 panels already use a shared `PanelContainer` wrapper (verified at `packages/spa/src/components/panels/PanelContainer.tsx`). The migration is "swap `<PanelContainer>` for `<Card><CardHeader>...</CardHeader>...</Card>`" — one wrapper per file, ~3-5 line diff. **The exception is `SkillHealth.tsx`** (252 lines, 8-state discriminated union including row-level click-to-expand) — its migration is wrapper-only too, but verifying the 8 states post-migration takes more test review than the others.

7. **Visual smoke test** — one-shot Playwright script under `packages/spa/scripts/screenshot.mjs` invoked manually (and post-Wave-5 in CI as a stretch goal). **No new test framework**; Playwright runs as a transient `npx --package=playwright -- playwright …` invocation, not a project dep, sidestepping the "no third-party JS additions" constraint. Output → `.planning/phases/DASH-05.1-…/refs/after-shell.png` per UI-SPEC §11 AC-08.

8. **Impeccable critique gate (preview)** — `impeccable:critique` is a Claude Code skill invoked from a chat (`/impeccable:critique`), not a CLI subprocess. It accepts a URL + viewport in its prompt; auth headers via Playwright's storage-state pattern aren't needed because the local dev server is open. Phase 5.1 closes with a manual `/impeccable:critique http://localhost:5174/projects/agenticapps-dashboard 1440x900` invocation; Phase 6 (POLISH-04) wires it as a hard gate.

**Primary recommendation:** Land Wave 0 (tokens + primitives + Inter import) as a single PR with no shell changes, then split Waves 1-5 across plans. Build AppShellV2 behind `VITE_APPSHELL_V2`, migrate route-by-route, and remove the flag in the same commit that deletes `AppShell.tsx`. Every panel migration is a wrapper swap; preserve test files unchanged where possible.

### Pattern 1: Tailwind 4 Token Strategy

**[VERIFIED: tailwindcss.com/docs/theme — separate-file pattern is explicitly recommended for "monorepo setups" and "team collaboration".]**

The Tailwind v4 idiomatic mechanism is the `@theme` directive in CSS, not `tailwind.config.ts`. The doc says: *"@theme requires explicit, top-level definition (not nested in selectors)"* and recommends splitting tokens into a separate file imported via `@import`.

**`packages/spa/src/styles/tokens.css` (NEW)**:
```css
/*
 * Phase 5.1 design tokens — single source of truth.
 * UI-SPEC §2-4 verbatim. AC-04, AC-05.
 */

@theme {
  /* Backgrounds — warm paper */
  --color-bg-app:        #FAFAF7;  /* main content area */
  --color-bg-sidebar:    #F8F6F3;  /* sidebar */
  --color-bg-card:       #FFFFFF;  /* cards on the warm bg */
  --color-bg-card-hover: #FAFAF7;

  /* Text */
  --color-text-primary:   #1F1B2E;  /* deep aubergine */
  --color-text-secondary: #6B6478;
  --color-text-tertiary:  #9C95A8;

  /* Accent — purple */
  --color-accent:           #6B46C1;
  --color-accent-hover:     #5B3BA8;
  --color-accent-bg:        #F2EBFA;
  --color-accent-bg-strong: #6B46C1;

  /* Borders */
  --color-border-subtle: #E8E5E0;

  /* Status */
  --color-status-success: #2E7D5B;
  --color-status-warning: #C2802B;
  --color-status-error:   #B53D3D;
  --color-status-info:    #5B6FA8;

  /* Card material */
  --shadow-card: 0 4px 12px rgba(0, 0, 0, 0.04);
  --radius-card: 12px;

  /* Typography */
  --font-sans: 'Inter Variable', ui-sans-serif, system-ui, -apple-system, sans-serif;
```

**`packages/spa/src/styles/global.css` (EDIT — minimal change)**:

### Pattern 3: TanStack Router Pathless Layout

**[VERIFIED: TanStack Router v1.169 docs — code-based pathless route uses `id` instead of `path`. Search results from tanstack.com/router/latest/docs/routing/code-based-routing confirmed 2026-05-09.]**

The current `router.tsx` mounts AppShell at `rootRoute`. **All 6 routes share the same shell**, including `/onboarding` and `/pair` — which is a bug per D-5.1-03. The migration both fixes the bug AND adds AppShellV2.

**`packages/spa/src/router.tsx` (REPLACEMENT shape)**:
```typescript
const rootRoute = createRootRoute({
  component: () => <Outlet />,    // bare — no shell here
})

// Pathless layout for paired routes — id only, no path.
// File-based equivalent would be `_appshell.tsx` (underscore prefix).
const appShellLayoutRoute = createRoute({
  getParentRoute: () => rootRoute,
  id: '_appshell',
  component: AppShellV2,          // mounts Sidebar + TopBar + <Outlet />
})

// Paired routes nested under appShellLayoutRoute (not rootRoute):
const indexRoute = createRoute({
  getParentRoute: () => appShellLayoutRoute,
  path: '/',
  beforeLoad: () => { /* unchanged */ },
}).lazy(() => import('./routes/index.lazy.js').then((m) => m.Route))

const settingsRoute = createRoute({
  getParentRoute: () => appShellLayoutRoute,
  path: '/settings',
}).lazy(/* unchanged */)

const helpRoute = createRoute({
  getParentRoute: () => appShellLayoutRoute,
  path: '/help',
}).lazy(/* unchanged */)

const projectsIdRoute = createRoute({
  getParentRoute: () => appShellLayoutRoute,
  path: '/projects/$projectId',
}).lazy(/* unchanged */)

// Standalone routes nested under rootRoute (no shell):
const onboardingRoute = createRoute({
  getParentRoute: () => rootRoute,
  path: '/onboarding',
}).lazy(/* unchanged */)

const pairRoute = createRoute({
  getParentRoute: () => rootRoute,
  path: '/pair',
  validateSearch: zodValidator(PairSearchSchema),
  errorComponent: pairErrorComponent,
}).lazy(/* unchanged */)

const routeTree = rootRoute.addChildren([
  appShellLayoutRoute.addChildren([indexRoute, settingsRoute, helpRoute, projectsIdRoute] as AnyRoute[]),
  onboardingRoute,
  pairRoute,
] as AnyRoute[])
```

**Why this works:**
- `createRoute({ id: '_appshell' })` creates a pathless layout (the `id` field replaces `path`; URL is unaffected).
- `appShellLayoutRoute.component = AppShellV2` mounts `<Sidebar />` + `<TopBar />` + `<Outlet />` for nested routes.
- `/onboarding` and `/pair` skip the layout entirely — their `getParentRoute` points back to `rootRoute`.
- Type inference still works (TanStack Router v1.169 supports this pattern fully).
- Existing `RepairBanner` + `CommandPalette` components currently mounted in AppShell move into AppShellV2 (or stay in `rootRoute`'s component if they should appear on `/onboarding` too — planner decides; default: move to AppShellV2 since unpaired surfaces don't need a re-pair banner).

### Pattern 5: Component Primitive Pattern (no `cn()`, no CVA)

**Codebase precedent**: `PanelContainer.tsx` already uses inline `className` composition with conditional flags (`stale`, `unreachable`). New primitives follow the same shape.

**`packages/spa/src/components/ui/Pill.tsx` (template for variant pattern)**:
```typescript
import React from 'react'

export type PillVariant = 'neutral' | 'accent' | 'success' | 'warning' | 'error'

const VARIANT_CLASSES: Record<PillVariant, string> = {
  neutral: 'bg-bg-card-hover text-text-secondary',
  accent:  'bg-accent-bg text-accent',
  success: 'bg-status-success/10 text-status-success',
  warning: 'bg-status-warning/10 text-status-warning',
  error:   'bg-status-error/10 text-status-error',
}

export interface PillProps {
  variant?: PillVariant
  children: React.ReactNode
}

export function Pill({ variant = 'neutral', children }: PillProps): React.JSX.Element {
  return (
    <span
      className={`inline-flex items-center rounded-md px-2 py-0.5 text-xs font-medium ${VARIANT_CLASSES[variant]}`}
    >
      {children}
    </span>
  )
}
```

**`packages/spa/src/components/ui/Card.tsx` (template for compound primitive)**:
```typescript
import React from 'react'

export interface CardProps {
  children: React.ReactNode
  className?: string  // escape hatch for grid placement only — not for color/spacing overrides
}

export function Card({ children, className = '' }: CardProps): React.JSX.Element {
  return (
    <section
      className={`rounded-[12px] bg-bg-card p-6 shadow-card ${className}`}
    >
      {children}
    </section>
  )
}

export interface CardHeaderProps {
  icon?: React.ReactNode      // lucide-react icon, 16px
  label: string
  helper?: string
  action?: React.ReactNode
}

export function CardHeader({ icon, label, helper, action }: CardHeaderProps): React.JSX.Element {
  return (
    <header className="flex items-center gap-2">
      {icon}
      <h2 className="text-lg font-semibold text-text-primary">{label}</h2>
      {helper && <span className="ml-auto text-sm text-text-tertiary">{helper}</span>}
      {action && <span className="ml-auto">{action}</span>}
    </header>
  )
}
```

**Why no `cn()`:**
- Codebase has zero usage today (`grep -rn "cn(" packages/spa/src` → 0 matches).
- Adding a helper saves ~5 chars per call, but every primitive only has 1-2 conditional class points. Net cognitive load goes UP.
- Stack lock forbids `clsx` and CVA additions.

### Anti-Patterns to Avoid

- **`tailwind.config.ts` `theme.extend` for new tokens** — v4 deprecates JS config; mixing the two is "schema drift" for design tokens. Stick to CSS `@theme`.
- **`<link href="https://fonts.googleapis.com/...">` for Inter** — third-party network dependency, breaks offline dev, violates "no analytics, no telemetry" spirit (Google Fonts logs IPs).
- **A `cn()` utility imported across 14 primitives** — saves keystrokes, costs cognitive load. The codebase has gone without it for 5 phases; don't break the streak for a stylistic preference.
- **Mounting AppShellV2 inside `rootRoute` and AppShell inside the pathless layout** — inverted. AppShell is the legacy shell that disappears in Wave 5; the layout route should host AppShellV2 from Wave 1 onward. The flag controls which sub-tree is in the route tree, not which component mounts where.
- **A new test framework for visual smoke** — Playwright as a transient `npx` invocation is sufficient. Don't introduce `@playwright/test` as a dep just to run one screenshot script.
- **Skeleton-shimmer loading states in any new primitive** — explicitly forbidden by D-5.1-10. Use static "Loading…" text or a single static `Loader` lucide icon.
- **Animation on sidebar item hover** — D-5.1-10 says hover states ONLY (no transitions). `bg-accent-bg` swap on hover is fine; `transition-all` is not.

## Common Pitfalls

### Pitfall 1: Tailwind 4 utility-class generation from `@theme` requires `--color-` prefix

**What goes wrong:** Author writes `--bg-app: #FAFAF7;` in `@theme {}` and expects `bg-bg-app` utility. Class doesn't exist; build silently produces no error but the utility is missing.

**Why it happens:** Tailwind v4 maps `@theme` namespaces to utility prefixes. `--color-*` → `bg-*`/`text-*`/`border-*`/`fill-*`/`stroke-*`. `--font-*` → `font-*`. `--text-*` → `text-*` (font-size). `--shadow-*` → `shadow-*`. `--radius-*` → `rounded-*`. Plain `--bg-app` lives outside any namespace and produces NO utility — only a CSS variable.

**How to avoid:** UI-SPEC §2 uses naming like `--bg-app`. The implementation MUST translate to `--color-bg-app` to get the utility. Document this in `tokens.css` comments. Fall-back: reference via `var(--bg-app)` in arbitrary value `bg-[var(--bg-app)]` — works, but verbose. Tokens with namespaces is cleaner.

**Warning signs:** Build succeeds but `<div className="bg-bg-app">` renders white instead of warm paper. DevTools Inspector shows the class in DOM but no CSS rule.

**Source:** [VERIFIED: Tailwind v4 docs at tailwindcss.com/docs/theme — namespace mapping table.]

### Pitfall 2: TanStack Router type inference breaks if pathless route has no `component`

**What goes wrong:** Add `appShellLayoutRoute` with `id: '_appshell'` but forget to set `component`; child routes don't render the shell because the layout route has no UI to wrap them.

**Why it happens:** Pathless route is a pure structural node. Without a `component` prop, it renders nothing (not even an `<Outlet />`).

**How to avoid:** `createRoute({ getParentRoute: () => rootRoute, id: '_appshell', component: AppShellV2 })` — `component` is mandatory for layout routes. AppShellV2 must render `<Outlet />` somewhere in its tree.

**Warning signs:** Routes resolve (URL changes) but page is blank. React DevTools shows `RootRoute > _appshell > IndexRoute` but no AppShellV2 in the tree.

### Pitfall 3: Inter Variable font-family name has a SPACE

**What goes wrong:** Author writes `font-family: Inter, sans-serif;` and expects the variable to load. Browser falls back to system Inter (or sans-serif) because the package registers under `'Inter Variable'`, not `'Inter'`.

**Why it happens:** `@fontsource-variable/inter` registers the family name as `Inter Variable` to disambiguate from `@fontsource/inter` (static weights, family name `Inter`). Co-installing both would conflict.

**How to avoid:** In `tokens.css`: `--font-sans: 'Inter Variable', ui-sans-serif, system-ui, -apple-system, sans-serif;` — note the **single quotes** and the word **Variable**. Tailwind's `font-sans` utility automatically wraps the value, no custom utility needed.

**Warning signs:** DevTools Computed tab shows `font-family: Inter Variable, ...` but Rendered Fonts panel shows "Inter (Local Fallback)" or "system-ui" — means the WOFF2 didn't load or the family name doesn't match.

**Source:** [CITED: fontsource.org/docs/getting-started/install — "@fontsource-variable" package family naming convention.]

### Pitfall 4: Vite env var truthiness — `VITE_APPSHELL_V2=true` ≠ `=1`

**What goes wrong:** Author writes `if (import.meta.env.VITE_APPSHELL_V2)` and tests with `VITE_APPSHELL_V2=true`. Works locally. Then a teammate sets `VITE_APPSHELL_V2=false` and the legacy shell never shows because `'false'` is truthy.

**Why it happens:** Vite stringifies all env vars at build time. `'false'` is a non-empty string → truthy.

**How to avoid:** Strict equality: `import.meta.env.VITE_APPSHELL_V2 === '1'`. Document the contract in `.env.example`: `VITE_APPSHELL_V2=1` (only `1` activates V2).

**Warning signs:** V2 appears when it shouldn't, or vice versa, depending on the literal string set.

### Pitfall 5: `@fontsource-variable/inter` import order matters

**What goes wrong:** Author imports `@fontsource-variable/inter` AFTER `./styles/global.css`. Brief flash where body uses fallback before Inter loads (FOUT — Flash of Unstyled Text).

**Why it happens:** Fontsource appends `@font-face` rules at the head of the document via the import side-effect. If the CSS that consumes `'Inter Variable'` is parsed first, it doesn't have the `@font-face` rule yet and uses the fallback. Once the import runs, the rule is added but the browser may not re-flow until a paint cycle later.

**How to avoid:** Import the font FIRST in `main.tsx`:
```typescript
import '@fontsource-variable/inter'   // FIRST
import { StrictMode } from 'react'
// …
import './styles/global.css'           // AFTER
```

**Warning signs:** First page paint shows system font, snaps to Inter ~50ms later. Visible during cold dev reload; may not appear in production build (which inlines critical CSS).

### Pitfall 6: Vite cache holds old `tokens.css` across edits

**What goes wrong:** Author edits `tokens.css`, sees no change in the dev server. Reverts edit, sees no change. Concludes the file isn't being read.

**Why it happens:** Vite caches CSS imports in `packages/spa/node_modules/.vite/`. Tailwind v4's plugin sometimes misses CSS-only file changes that don't hit the JS dependency graph.

**How to avoid:** Restart the dev server with `pnpm --filter @agenticapps/dashboard-spa dev --force`. Better: hot-reload edits to `global.css` (which `@import`s `tokens.css`) — the chain triggers HMR.

**Warning signs:** Token edit not visible; Network tab shows 304 Not Modified on the CSS bundle.

### Pitfall 7: Migration order matters for the test suite

**What goes wrong:** Author migrates `Card` primitive in Wave 0, then in Wave 1 migrates `PanelContainer` consumers to use `Card`. Tests for `PanelContainer` panels pass throughout Wave 0; in Wave 1 they fail because the new `Card` doesn't expose the same DOM attributes (`aria-labelledby={\`${panelId}-title\`}`).

**Why it happens:** `PanelContainer` ships `aria-labelledby` and `<header>` semantics; tests assert on these. `Card` + `CardHeader` may use different ARIA hooks.

**How to avoid:** Wave 0 ships `Card` + `CardHeader` with **the same a11y contract as `PanelContainer`** — `aria-labelledby` linking to the heading's `id`. New tests assert the same. Panel-level tests then continue to pass when their wrapper swaps from `PanelContainer` to `Card`. Where the contracts can't match, update the panel tests in the same commit as the wrapper swap (preferred over swapping wrappers in one wave and updating tests in another — that leaves the suite red between waves).

**Warning signs:** Whole-panel tests fail with "Unable to find a label associated with…" after a wrapper swap.

### Pitfall 8: `gap-6` between cards inside columns vs `gap-x-6` between columns

**What goes wrong:** Author uses `grid grid-cols-3 gap-6`. Card-to-card vertical gap inside a column is correct. Column-to-column horizontal gap is also 24px. But the existing layout has `flex flex-col gap-4` inside the column already (16px gap between panels) — applying `gap-6` on the grid then `gap-4` on the inner `flex` produces inconsistent rhythm.

**Why it happens:** The current `SingleProjectView.tsx` uses `flex flex-col gap-4` per column. UI-SPEC §1 says card gap is 24px (`gap-6`). The existing inner `gap-4` (16px) needs to update to `gap-6` (24px) at the same time the grid's `gap-6` lands.

**How to avoid:** When migrating `SingleProjectView`, change BOTH the grid's gap AND the column's `flex-col gap-4` → `flex-col gap-6`. Verify with DevTools that vertical card gaps match horizontal column gaps.

**Warning signs:** Visual rhythm feels "tight" in columns and "loose" between columns, or vice versa.

## Visual Smoke Test Strategy

UI-SPEC §11 AC-08 demands a Playwright screenshot at `refs/after-shell.png`. Lightest path:

**Approach: One-shot Playwright via `npx --package=`**

1. **No project dep added.** Playwright runs as a transient `npx` invocation. Project's `node_modules` and `package.json` stay untouched.
2. Script lives at `packages/spa/scripts/screenshot.mjs` (see Example 4 above). Pure ESM, runs on Node 20 native.
3. Manual invocation:
   ```bash
   pnpm --filter @agenticapps/dashboard-spa dev &  # background
   sleep 3                                          # wait for server
   npx --package=playwright -- node packages/spa/scripts/screenshot.mjs
   ```
4. Output committed to `.planning/phases/DASH-05.1-…/refs/after-shell.png`. ~150kB PNG, fine for git.

**Why NOT add Playwright as dep:**
- Stack lock forbids unnecessary additions.
- Playwright pulls a 200MB browser binary into `node_modules/.cache/ms-playwright/` per developer.
- One screenshot per phase doesn't justify the install.

**Why NOT use a different tool (puppeteer, headless-chrome, jsdom):**
- jsdom can't render CSS layout → no real screenshot.

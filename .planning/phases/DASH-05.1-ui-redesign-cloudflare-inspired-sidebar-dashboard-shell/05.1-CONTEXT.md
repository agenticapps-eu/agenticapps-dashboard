# Phase 05.1: UI Redesign — Cloudflare-Inspired Sidebar Dashboard Shell - Context

**Gathered:** 2026-05-09
**Status:** Ready for UI design contract → variant exploration → planning
**Anchor:** User wants Cloudflare-dashboard feel — minimal, uncluttered, sidebar-driven navigation that scales as more functionality lands.

<domain>
## Phase Boundary

A **UI-only redesign** of the SPA shell and route layouts. No backend changes, no new features, no removed capabilities — every panel, route, query, and daemon contract that ships today continues to ship after 5.1. What changes is *the chrome around them*.

In scope:
- Replace the current top-header-only navigation with a **sidebar-driven shell** inspired by Cloudflare's dashboard surfaces.
- Restructure the four existing routes (`/`, `/projects/:id`, `/settings`, `/help`) inside the new shell.
- Establish a **design-system layer** (tokens, typography scale, spacing scale, component primitives) so future phases stop redefining one-offs.
- Polish the visual quality of all existing panels to fit the new shell — no anti-AI-slop regressions; ideally improve toward the Phase 6 impeccable ≥ 90 gate.
- Keep `/onboarding` and `/pair` deliberately out of the sidebar — those are pre-paired surfaces.

Out of scope (see Deferred):
- New panels, new routes, new daemon endpoints.
- Cmd+K command palette redesign (Phase 3 D-3-09 implementation kept; visual polish allowed).
- Theme system rework beyond the existing light/dark toggle.

</domain>

<decisions>
## Implementation Decisions

### Scope & Intent

- **D-5.1-01:** This phase ships chrome + design system, not features. Every panel that exists today must still exist after 5.1, with the same data and behavior. Acceptance: a smoke test on each route shows the panel populated with real data from a registered project.
  - **Why:** Without this rule, "redesign" silently turns into "rewrite" and the phase blows past its session budget.

- **D-5.1-02:** The redesign applies to **all paired routes** (`/`, `/projects/:id`, `/settings`, `/help`), not just the project view. Inconsistency between routes was already the strongest visual smell from a casual look at the Phase 5 screens.
  - **Why:** A sidebar shell that wraps only one route is worse than no sidebar — it makes the others look orphaned.

- **D-5.1-03:** `/onboarding` and `/pair` keep their existing standalone presentation (no sidebar). They are pre-paired surfaces; the sidebar implies a paired session.
  - **Why:** Phase 2 D-2-01 already locked the onboarding hero pattern; revisiting would be scope creep.

### Aesthetic Direction (the one decision the user has explicitly locked)

- **D-5.1-04:** Cloudflare dashboard aesthetic. Reference screenshots in `refs/`:
  - `cf-login.png` — minimal centered card, generous whitespace, orange accent reserved for branding only.
  - `cf-developers-pages.png` — three-column doc-style layout: 240px sidebar with collapsible sections + content + right TOC rail.
  - `cf-workers-playground.png` — workspace mode: 50px icon rail + main content + sub-panels.
  - **Common DNA:** white/near-white background, hairline borders, deep neutral-gray text, blue (#0073EC-ish) primary, brand color (orange in CF, **TBD** for us) reserved for the logo/identity stripe and *one* signature accent.
  - **Density:** medium-low. 24-32px section padding, 16-24px card padding. Anti-clutter is a discipline, not a style.
  - **Why:** This is the user's explicit ask. Other minimal-dashboard references (Linear, Vercel, Stripe) are similar enough that variants can lean adjacent without violating intent.

### Sidebar Direction (where variants get generated)

- **D-5.1-05:** Sidebar IS the primary navigation. The current top-header `Header` shrinks to a thin global utility bar (or disappears entirely; that's a variant question). Routes live in the sidebar; route metadata (e.g., "N projects · refreshed 3s ago") lives in a per-page header.
  - **Why:** User explicit ask: "I like the sidebar menu approach for this kind of dashboard so I can easily switch between functions."

- **D-5.1-06:** **Variants explored in `/design-shotgun`, not pre-decided here.** The shotgun step generates 3–5 distinct sidebar treatments rendered against real Phase 5 data. User picks the direction.
  - **Why:** User explicit ask: "show me different variants before going into implementation."
  - Variant axes the shotgun should explore:
    - **Width**: 50px icon-rail (Workers playground style) vs 240px text-rail (CF docs style) vs hybrid (collapsed-with-pin like VSCode).
    - **Depth**: flat list of routes vs grouped sections with collapsible headers.
    - **Project switching**: workspace switcher at top (Linear-style) vs project list inside the sidebar (CF accounts-style) vs separate `/` home page (current).
    - **Theme**: light only, dark only, or theme-toggleable; whether the sidebar uses a slightly darker neutral than main content.
    - **Accent color identity**: orange (CF echo), blue (most dashboards), purple/teal (differentiate), or **none** (pure neutral with one signature element).

### Components & Design System (the durable artifact)

- **D-5.1-07:** Establish a tokens file (`packages/spa/src/styles/tokens.css` or similar) with: color scale, type scale (text-xs through text-3xl with explicit line-heights), spacing scale (already on Tailwind defaults; just commit to using it), radius scale, shadow scale (probably 3 levels max), and a single z-index ladder.
  - **Why:** Phase 5 shipped solid panels but each phase reinvents margin/padding choices. This is the right phase to lock the system before Phase 6 polishes against it.

- **D-5.1-08:** Componentize: extract `Sidebar`, `SidebarItem`, `SidebarSection`, `PageHeader`, `Card`, `EmptyState`, `Pill`, `KbdShortcutHint` as proper components in `packages/spa/src/components/ui/`. Migrate existing panels to use them.
  - **Why:** Each panel currently rolls its own card/header/empty pattern. Consolidation is the only way density/whitespace stays consistent across panels.

- **D-5.1-09:** Typography: pick **one** display family + system stack body fallback. Cloudflare uses a custom geometric sans; we'll use **Inter** (variable, free, near-identical feel) as the explicit choice unless a shotgun variant proposes something stronger and the user picks it.
  - **Why:** Inter is the "no-regret" path. Variant-shopping fonts is a rabbit hole.

### Anti-AI-Slop Discipline (carries from Phase 3 D-43)

- **D-5.1-10:** No animations on route entry, sidebar open, or panel mount. Hover states only. Focus rings on keyboard navigation. No skeleton-shimmer (loading uses static spinner or simple text). No gradient backgrounds beyond at most one subtle accent stripe. No emoji as iconography (lucide-react only, already a dep).
  - **Why:** Phase 3 D-43 established this discipline. The redesign must not regress it. Animations and shimmers are how AI-generated dashboards usually telegraph "AI-generated."

### Implementation Approach

- **D-5.1-11:** Build new shell components first (Sidebar + AppShellV2 + PageHeader), wire them in alongside the current AppShell behind a dev-only toggle, migrate routes one at a time, then delete the old shell. Not a big-bang rewrite.
  - **Why:** Each route stays shippable mid-phase. If the session runs out of budget, we can land what's done and finish in a follow-up.

- **D-5.1-12:** All component work TDD per workflow contract. Existing route + panel tests must continue to pass against the new shell — that's the "no functional regression" gate.

### Claude's Discretion

- Exact icon-rail width if that variant wins (default: 56px = comfortable touch target).
- Exact accent color if a non-neutral wins (default within each variant: orange for "CF echo", blue for "industry standard").
- Whether the right-rail TOC pattern (CF docs) appears on `/help` and `/settings`. Default: yes on `/help`, no on others.
- Whether per-project sidebar shows recent commits as sub-items. Default: no — that's a Phase 7 idea.

</decisions>

<specifics>
## Specific Ideas

- **The Cloudflare DNA is restraint, not orange.** The shotgun should explore neutral-only variants alongside accent variants. Orange is CF's identity; ours can be different (or absent).
- **The sidebar is the IA, not a decoration.** If a variant adds a sidebar but the routes inside still each have their own navigation patterns, the variant is fighting itself. Every variant should make the sidebar the *only* primary navigation.
- **Real data in shotgun mocks.** The shotgun must render against the dashboard's actual Phase 5 state, not lorem-ipsum. Dogfooding a real registered project (this repo, the dashboard's own `.planning/`) gives the user a concrete decision-grade view.
- **The redesign is a chance to delete.** If a panel's chrome can be simpler or removed inside the new shell, do it. Don't preserve every separator and label by default.
- **Variants are NOT reskin only.** Different sidebar widths and grouping strategies change density and information architecture meaningfully. Don't ship 5 versions of "the same sidebar with different colors."

</specifics>

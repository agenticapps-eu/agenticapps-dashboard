### What landed

**Task 1: MaskedToken integration into ManualPairForm (D-6.1-03)**

When `ManualPairForm` mounts and `getPairing()` returns a non-null pairing, the token field renders as a `<MaskedToken value={token} label="Bearer token">` plus an "Edit" button instead of the plaintext `<input>`. User clicks Edit to switch to a standard input (auto-focused with the current value). Copy-to-clipboard works while masked because `<MaskedToken>`'s internal Copy button writes the actual `value` prop, not the displayed dots.

Implementation details:
- New state: `tokenEditMode` (boolean), `tokenInputRef` (auto-focus on flip), `initialTokenRef` (captures pre-populated token at mount).
- `showMaskedToken = (initial token was non-null) && !editMode && (current token still matches initial)`. If user types something different, the masked view doesn't apply.
- Input element preserves all existing semantics (validation, `aria-invalid`, `aria-describedby`, onBlur validation, `readOnly` during submit).
- Helper text reads "Token is masked. Click Reveal to see it, or Copy to copy without revealing. Click Edit to enter a new token." when in masked mode.
- Anti-AI-slop preserved: instant render swap (no transitions, no animations).

**Task 2: /settings page-level helper prose cap (D-6.1-01)**

`PageHeader`'s `helper` prop accepts `string` only and renders without a `max-width` cap. Refactored `/settings` to render helper prose as a separate `<p>` below `PageHeader` so we can apply `max-w-[75ch]` without modifying `PageHeader` (which would affect `/`, `/projects/:id`, and `/help` where the helper is metadata-style, not prose).

Also from Task 1's same-file edits: `max-w-[75ch]` applied to `ManualPairForm`'s `token-helper` and `agent-url-helper` `<p>` elements (multi-sentence prose qualifies per D-6.1-01).

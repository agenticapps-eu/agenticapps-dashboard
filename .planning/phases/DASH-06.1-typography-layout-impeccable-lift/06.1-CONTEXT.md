<decisions>
## Implementation Decisions

### Line-length policy

- **D-6.1-01:** `max-w-[75ch]` applies to **card body text + help/onboarding/settings prose only**. Explicitly: multi-sentence prose paragraphs in card bodies (CommitmentBlock descriptions, SkillHealth metadata explanations, panel "configure to enable" copy), all `<p>` content in /help shortcut explanations, /onboarding hero subtitle and steps, /settings descriptions and field hints. **Excluded:** card metadata (counts, IDs, status pills, timestamps), all headings (h1-h6), code blocks (`<pre>`, `<code>` blocks; inline `<code>` inherits container), kbd hints.
  - **Why:** Headings need to break out wider for visual hierarchy; constraining them produces awkward line breaks. Card metadata is tabular-feel data, not prose. Code blocks (e.g., command examples in /help, JSON in CF Access doc) need horizontal room. Targeting prose specifically lifts both Typography (line-length is a Nielsen-typography concern) and Layout (75ch is the optimal-readability default) without over-constraining the rest.
  - **How to apply:** Audit each route for prose containers and apply `max-w-[75ch]` (Tailwind's arbitrary-value syntax ‚Äî already supported, no token addition needed). Planner picks whether to wrap in a shared `<Prose>` primitive or apply inline.

### Integration panel progressive disclosure

- **D-6.1-02:** Panels in "configure to enable" empty state collapse by default; data-bearing panels stay expanded. The 5 panels under disclosure: `IntegrationsHealth`, `ObservabilityHealth`, `SecurityStatus`, `SkillHealth`, `VerificationStatus` (all in `packages/spa/src/components/panels/`). Trigger: click anywhere on the panel header to expand. **No localStorage persistence** ‚Äî state resets per page-load.
  - **Why:** The "spread thin" verdict from 06-06's measurement comes from 5 mostly-empty panels eating vertical real estate on /projects/:id. Collapsing the empty ones reduces visual density (Layout sub-score) without hiding real data. Per-page-load reset keeps state simple ‚Äî no "where did my expansion go" surprises, no localStorage to reason about.
  - **How to apply:** Extend `PanelContainer.tsx` with optional `defaultCollapsed` prop + click-to-toggle on header. Each panel's render decides `defaultCollapsed = isEmpty(data)`. Aria: `aria-expanded` on the header button, `aria-controls` referencing the panel body. Planner picks chevron icon style (must be tokens-only ‚Äî no animation per D-43; instant state toggle).

### Token masking + sensitive-data display in /settings

- **D-6.1-03:** Bearer tokens **masked by default** in /settings; click "show" reveals + **auto-hides after 5 seconds**. Project paths and agent IDs are NOT masked (they aren't secrets). Copy-to-clipboard button is **always visible** ‚Äî copy works without revealing first.
  - **Why:** Bearer tokens are the only real secret. Auto-hide protects against shoulder-surfing if user walks away from the screen. Copy-without-reveal is the safer pattern (clipboard is readable; revealing first adds friction without security gain). Path masking would create false-security theater.
  - **How to apply:** New `<MaskedToken>` primitive in `packages/spa/src/components/ui/` (mirrors existing primitive patterns). Internally a small state machine: `masked ‚Üí revealed (5s timer) ‚Üí masked`. `aria-label` describes the masked content (e.g., "Bearer token, masked"). Copy button uses `navigator.clipboard.writeText` against the actual token (not the displayed dots). Tests cover: timer auto-hide, manual re-mask, copy-while-masked, ARIA states. Anti-AI-slop: instant transitions only ‚Äî no fade between states.

### Accessibility scope: ARIA + empty states

- **D-6.1-04:** Comprehensive ARIA on OBSERVE-section navigation + numbered-steps empty state on /pair.
  - ARIA additions:
    - `aria-current="page"` on the active sidebar route (`SidebarItem` / `SidebarSubItem`)
    - `aria-label` on icon-only OBSERVE items (so screen readers announce "Skills section" not just "icon button")
    - `aria-live="polite"` on daemon-status display (TopBar StatusPill + any inline daemon-state announcers like SchemaDriftState/DaemonUnreachableState)
  - /pair empty-canvas treatment: numbered steps in plain text. NO hero illustration (anti-AI-slop). NO interactive walkthrough. Just: 1. "Open the printed pair URL". 2. "Approve the pairing in your terminal". 3. "Wait for the dashboard to populate".
  - **Existing OnboardingHero on `/` stays as-is** ‚Äî Phase 5.1 approved it; not in scope.
  - **Why:** Accessibility is part of the impeccable composite (Nielsen heuristic 1: visibility of system status; SC 2.4.3 focus order; SC 4.1.2 name/role/value). Lifting accessibility lifts the Hierarchy and overall composite scores. Numbered steps on /pair give the empty canvas hierarchy without scope creep ‚Äî anti-AI-slop discipline rules out hero art and walkthroughs.
  - **How to apply:** Edit `Sidebar.tsx`/`SidebarItem.tsx`/`SidebarSubItem.tsx` for `aria-current` + `aria-label`; edit `TopBar.tsx` (or wherever StatusPill renders) for `aria-live`. Edit `pair.lazy.tsx` (or split into a `PairEmptyState.tsx` component) for numbered-steps content. Add a11y tests: `getByRole`, `getByLabel`, axe-core integration if not already wired.

### Claude's Discretion

- Exact `<Prose>` primitive vs inline `max-w-[75ch]` application ‚Äî planner chooses based on usage frequency.
- Chevron icon style for panel disclosure trigger ‚Äî must be tokens-only, instant state change (no rotation animation per D-43).
- Whether `<MaskedToken>` lives in `ui/` or `components/` ‚Äî planner picks per existing primitive layout.
- Planner picks how to thread `aria-live` through React state without re-announcing on every keystroke (debouncing pattern, etc.).
- Test coverage approach: rely on existing `@testing-library/jest-dom` matchers vs add `axe-core` for automated a11y assertions ‚Äî planner picks based on cost/value.
- Whether to extract a shared `<EmptyCanvas>` primitive for /pair or inline the numbered-steps markup ‚Äî depends on whether other routes might need it.
- Numbered-step copy on /pair: planner drafts the actual wording (technical writing within the 3-step structure).

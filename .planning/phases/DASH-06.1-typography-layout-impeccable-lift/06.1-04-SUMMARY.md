# Phase 06.1 Plan 04: Accessibility Scope — ARIA on Sidebar + TopBar + /pair Empty Canvas

ARIA semantics on sidebar navigation (aria-current, aria-label on disabled items), single canonical aria-live="polite" region in TopBar, and numbered-steps empty canvas on /pair during pairing state — closing the Accessibility and Layout gaps from the 06-06 post-polish baseline.

## What Was Built

### Task 1: ARIA on Sidebar

**SidebarItem** — The active `<Link>` now carries `aria-current="page"` when `isActive` is true; when inactive the attribute is omitted entirely (not set to "false" — semantic difference per WCAG SC 4.1.2). Applied via `aria-current={isActive ? 'page' : undefined}`.

**SidebarItemDisabled** — All 3 disabled OBSERVE buttons (Skills, Health, Reviews) now carry `aria-label={\`${label} section, available in Phase 6\`}` alongside the existing `title="Available in Phase 6"`. Screen readers announce purpose rather than just "button".

**SidebarSubItem** — Same `aria-current` pattern as SidebarItem applied to the project sub-nav links.

The Link mock in both test files was updated to forward the `aria-current` prop with the correct `React.AriaAttributes['aria-current']` type.

### Task 2: TopBar aria-live Region

The conditional tags Pill row and phase StatusPill in TopBar are now wrapped in a single `<div role="status" aria-live="polite" className="flex items-center gap-2">`. The wrapper is rendered unconditionally (always present, even when empty on non-project routes) so the assistive technology registers the live region at mount before any update fires — the recommended ARIA-live-region registration pattern.

TB10 was updated: previously it used `queryByRole('status')` to detect HelpOverlay presence/absence. With the always-present status wrapper, that approach broke. TB10 now checks for the HelpOverlay via `queryByRole('button', { name: /got it/i })` instead.

Exactly 1 `aria-live` region in the file. `aria-live="assertive"` is absent from the entire SPA source tree. SchemaDriftState and DaemonUnreachableState unchanged (no double-announce).

### Task 3: /pair Numbered-Steps Empty Canvas

The `pairing` branch in `pair.lazy.tsx` was replaced with a structured empty canvas:

- `<h1>Pairing in progress</h1>` heading
- `<p className="max-w-[75ch]">Hold tight...</p>` subtitle  
- `<ol>` with 3 semantic `<li>` items (browser provides numerals — no decorative SVG):
  1. Open the printed pair URL in this browser.
  2. Approve the pairing in your terminal.
  3. Wait for the dashboard to populate.
- `<p role="status" aria-live="polite">Connecting to agent…</p>` — preserves existing assertions

No `<img>`, `<svg>`, or `<details>` in the pairing canvas. Anti-AI-slop discipline holds.

The `failed` branch prose also gained `max-w-[75ch]` (D-6.1-01 carry-forward — small enough to bundle here).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Link mock type too broad for TypeScript strict mode**
- **Found during:** Task 1 GREEN — typecheck after implementing aria-current
- **Issue:** The Link mock in SidebarItem.test.tsx and SidebarSubItem.test.tsx typed `'aria-current'?: string`, but the actual DOM `aria-current` attribute is a union type `boolean | 'date' | 'time' | 'true' | 'false' | 'page' | 'step' | 'location' | undefined`. TypeScript rejected `string` as incompatible.
- **Fix:** Changed mock parameter type to `React.AriaAttributes['aria-current']`
- **Files modified:** SidebarItem.test.tsx, SidebarSubItem.test.tsx
- **Commit:** c95fa3b

**2. [Rule 1 - Bug] TB10 test broken by always-present role=status wrapper**
- **Found during:** Task 2 RED — realized the new `role="status"` div conflicts with TB10's `queryByRole('status')` assertion
- **Issue:** TB10 expected `queryByRole('status')` to return null before HelpOverlay is shown, but the always-present TopBar wrapper also carries `role="status"`.
- **Fix:** Updated TB10 to check for HelpOverlay via `queryByRole('button', { name: /got it/i })` instead — tests the same behavior (overlay visible/hidden) without relying on the status role count.
- **Files modified:** TopBar.test.tsx
- **Commit:** 50b5286 (part of RED commit), d3f1906 (GREEN)

**3. [Rule 1 - Bug] Non-null assertion needed on NodeList index access**
- **Found during:** Task 2 GREEN — typecheck on liveRegions[0].getAttribute()
- **Issue:** `NodeList` index access returns `Element | undefined` under strict null checks.
- **Fix:** Used non-null assertion `liveRegions[0]!` after the `toHaveLength(1)` assertion confirms presence.
- **Files modified:** TopBar.test.tsx
- **Commit:** d3f1906

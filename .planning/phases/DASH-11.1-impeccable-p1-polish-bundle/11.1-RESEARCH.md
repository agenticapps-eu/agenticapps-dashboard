### B4 — `usePageHeaderHeight` hook contract

```ts
// usePageHeaderHeight.ts
export function usePageHeaderHeight(ref: React.RefObject<HTMLElement>): void {
  useEffect(() => {
    if (!ref.current || typeof ResizeObserver === 'undefined') return
    const el = ref.current
    const ro = new ResizeObserver(([entry]) => {
      const h = entry.contentRect.height
      document.documentElement.style.setProperty('--ph-h', `${Math.round(h)}px`)
    })
    ro.observe(el)
    return () => ro.disconnect()
  }, [ref])
}

// :root in tokens.css
:root {
  --ph-h: 56px; /* fallback before ResizeObserver fires */
}
```

PageHeader binds the ref:

```tsx
const ref = useRef<HTMLDivElement>(null)
usePageHeaderHeight(ref)
return <div ref={ref} className={`mb-6 flex flex-col gap-1${stickyClasses}`}>...</div>
```

CoverageFamilySection consumes:

```tsx
<header className="sticky top-[calc(var(--ph-h)-1.5rem)] z-20 ...">
  ...
</header>
<th className="sticky top-[calc(var(--ph-h)+1.5625rem)] z-10 ...">
  ...
</th>
```

The `-1.5rem` matches PageHeader's `top-[-1.5rem]` offset (the sticky-floor compensation). The `+1.5625rem` (25px) gives the column headers room for the family-header (49px) — derived as `family-header-height + 1px-border - PH-bottom-offset`.

## §C — Toast primitive (P1 #3)

### C1 — Alternatives considered (placement + queue)

| Option | Position | Stack | Animation | Verdict |
|---|---|---|---|---|
| **A. Top-right portal, single-slot** (CHOSEN) | `fixed top-16 right-4` | 1 toast — new replaces old | opacity 150ms | **Accepted as D-11.1-06** |
| B. Bottom-center portal, FIFO queue of 3 | `fixed bottom-4 left-1/2` | 3 toasts stacked, FIFO eviction | slide-up + fade | Rejected. Competes with mobile virtual keyboard. Stack-management complexity. |
| C. Inline confirmation badge per CTA | rendered next to clicked button | N/A — per-CTA local state | fade in/out | Rejected. Forces every clipboard call site to manage local state — defeats the "scoped primitive" goal. |
| D. Top-banner full-width notification | `fixed top-0 left-0 right-0` | 1 banner | slide down | Rejected. Too prominent for clipboard feedback; competes with TopBar. |

### C2 — API design rationale (D-11.1-07)

Intentionally minimal:

- `show(input)` — fire-and-forget. No promise chain. No callback. No id-return. Composable via `useCallback`.
- `ToastVariant` is 2-valued (success / error) and not extended. If a future surface needs `info` or `warning` toasts, add then — not now.
- `duration: 0` reserved sentinel for "sticky until dismiss" — not used in Phase 11.1, but the type allows it so the API doesn't need re-baselining.
- No imperative dismiss API in v1 — toasts dismiss themselves on duration timeout. If a future panel needs `dismiss()`, add it then.

### C3 — Provider mount strategy (D-11.1-09)

`AppShellV2` (or the route-shell root) wraps with `<ToastProvider>`. Pre-pair routes (login, pair) are explicitly skipped — they have no clipboard writes and shouldn't carry the context overhead. Single mount point keeps the portal in one DOM location.

### C4 — Portal target

Two options:

| Option | Notes |
|---|---|
| `document.body` | Simplest. Risk: React StrictMode double-render in tests creates two `<div>`s. Mitigation: idempotent root creation. |
| Dedicated `<div id="toast-root">` inside AppShellV2 | Test ergonomics cleaner — root is auto-managed by React. But requires the toast to position above all other AppShellV2 children via z-index (already handled by `--z-toast: 2000`). |

Planner picks based on test runs.

## §D — Token swap (P1 #4)

### D1 — Background contrast computation

Using WCAG 2.1 relative luminance formula:

```
L = 0.2126 * R' + 0.7152 * G' + 0.0722 * B'
where R',G',B' are the gamma-decoded channel values:
  c' = c/255 ≤ 0.03928 ? c/255/12.92 : ((c/255 + 0.055)/1.055) ** 2.4
```

Contrast ratio:
```
CR = (L_lighter + 0.05) / (L_darker + 0.05)
```

### D2 — Candidate analysis (full computation)

```
Background app-bg #FAFAF7 → L ≈ 0.95779
Background sidebar-bg #F8F6F3 → L ≈ 0.94717

Candidates (foreground):
  #807A92 (current)     L ≈ 0.20305 → CR vs app-bg ≈ 3.99:1 ; vs sidebar-bg ≈ 3.95:1  ❌
  #7A7388               L ≈ 0.18250 → CR vs app-bg ≈ 4.40:1 ; vs sidebar-bg ≈ 4.35:1  ⚠ borderline
  #75708A               L ≈ 0.16997 → CR vs app-bg ≈ 4.69:1 ; vs sidebar-bg ≈ 4.64:1  ✓ chosen
  #6B6580               L ≈ 0.14572 → CR vs app-bg ≈ 5.32:1 ; vs sidebar-bg ≈ 5.27:1  ✓ but tier gap closes
  #605A72               L ≈ 0.11703 → CR vs app-bg ≈ 6.27:1 ; vs sidebar-bg ≈ 6.20:1  ✓ but indistinguishable from secondary

Secondary token (reference):
  #6B6478               L ≈ 0.15154 → CR vs app-bg ≈ 5.18:1 ; vs sidebar-bg ≈ 5.13:1
```

### D3 — Why `#75708A` (not the IMPECCABLE-doc-suggested `#6B6580`)

The IMPECCABLE finding (line 158) suggested `#6B6580`. That value clears 4.5:1 with headroom (~5.32:1) BUT compresses the tertiary tier against the secondary tier:

- Current: tertiary @ 3.95-3.99, secondary @ 5.13-5.18 → ~1.3:1 inter-tier ratio
- With `#6B6580`: tertiary @ 5.27-5.32, secondary @ 5.13-5.18 → ~1.02:1 inter-tier ratio (visually identical)
- With `#75708A`: tertiary @ 4.64-4.69, secondary @ 5.13-5.18 → ~1.10:1 inter-tier ratio (visible delta preserved; tier system holds)

The design system's text tiers signal hierarchy (primary > secondary > tertiary). Compressing tertiary up to secondary collapses the hierarchy. `#75708A` clears AA AND preserves the visible tier delta.

### D4 — Test contract (the invariant lock)

```ts

import { contrastRatio } from '../lib/contrast'

const APP_BG = '#FAFAF7'
const SIDEBAR_BG = '#F8F6F3'

describe('Token contrast invariant (WCAG 2.1 AA body text)', () => {
  const tiers = {
    primary:   { hex: '#1F1B2E', minRatio: 13.0 },
    secondary: { hex: '#6B6478', minRatio: 4.5 },
```

This test fires on every CI run and lights up if any future token edit regresses contrast. Closes the v1.0.1 → v1.1 regression class (where the v1.0.1 bump only cleared 3:1, not 4.5:1).

## §E — Sequencing analysis (for planner)

The four P1s have minimal cross-dependency:

```
P1 #4 (token swap)         — totally independent; can land first, alone
P1 #3 (Toast primitive)    — independent of P1 #1/#2; sequential dep: Toast.tsx → wire 6 call sites
P1 #2 (sticky toolbar)     — depends on P1 #1 ONLY because both touch CoverageFamilySection.tsx
                             (column-header `top-` adoption from var(--ph-h))
P1 #1 (column-width lock)  — independent; affects only CoverageFamilySection.tsx + CoverageRow.tsx
```

Suggested wave structure:

- **Wave 0 (sequential, foundational):** `lib/contrast.ts` + tests + verify-contrast.test.ts + token swap. Tiny — single commit, no parallel needed.
- **Wave 1 (parallel, no file overlap):**
  - Plan A: `coverageColumns.ts` + `CoverageFamilySection.tsx` `<colgroup>` + `CoverageRow.tsx` column-track adoption.
  - Plan B: `Toast.tsx` primitive + tests.
  - Plan C: `usePageHeaderHeight.ts` + hook test + PageHeader integration.
- **Wave 2 (sequential, depends on Wave 1):**
  - Plan D: `CoveragePage.tsx` move CoverageToolbar into PageHeader children + `CoverageFamilySection.tsx` switch `top-` constants to `var(--ph-h)` expressions.
  - Plan E: Wire 6 clipboard call sites to `useToast()`. Includes ToastProvider mount in AppShellV2.

5 plans total. Wave 0 + Wave 1 + Wave 2 ≈ 3 parallel cycles.

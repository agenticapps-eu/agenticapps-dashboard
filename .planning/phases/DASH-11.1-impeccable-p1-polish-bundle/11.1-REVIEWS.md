# Cross-AI Plan Review — Phase 11.1

Two independent AI reviewers (Codex via OpenAI, Gemini via Google) read all 6 plans + CONTEXT + RESEARCH + UI-SPEC + the authoritative `11-IMPECCABLE.md` source and produced structured feedback.

**Gating verdict:** the 4 HIGH-severity codex concerns are blocking — they describe test harnesses and component APIs that would NOT compile against the current codebase. Plans 04 and 06 are revised before execute kicks off. Gemini's LOW-severity suggestions are folded into the same revisions.

---

## Codex Review

**Risk Assessment: MEDIUM**

### Summary

> The phase plan is directionally strong: scope is tight, decisions are locked, the wave split mostly matches the real dependency graph, and the phase stays inside the intended boundary of `/coverage` polish without dragging daemon or API work into it. The main weakness is test quality. Several RED/GREEN steps verify source shape or class strings rather than user-visible behavior, and two tasks are not executable as written against the current codebase, especially `usePageHeaderHeight` testing and the `AppShellV2` provider test.

### Strengths

- Phase boundary disciplined (SPA/UI/token only — no daemon)
- Decomposition sensible: 01 establishes contrast first; 03+04 create primitives before 05+06 wires them
- Repeated sticky-stack regression verification protects Phase 11
- `coverageColumns.ts` + `contrast.ts` at the right level of abstraction
- Toast intentionally restrained — matches calm-instrument register

### Concerns (with severity)

**HIGH**
- **C-CODEX-H1** — Plan 11.1-04 Task 1 `usePageHeaderHeight` hook test is not executable as written. The proposed `renderHook(() => { const ref = useRef(...); usePageHeaderHeight(ref); return ref })` mutates `ref.current` after render — does NOT reliably retrigger the hook effect. Needs a real mounted component + callback-ref harness.
- **C-CODEX-H2** — Plan 11.1-06 Task 1 assumes `AppShellV2` accepts a `children` prop, but current `packages/spa/src/components/AppShellV2.tsx:19` renders `<Outlet />`. Test `<AppShellV2 children={<Probe />} />` will NOT compile.
- **C-CODEX-H3** — IMP-03's "visible toast within 200ms" requirement is not actually locked by tests. Plan 03 proves the primitive renders; Plan 06 proves wiring; neither enforces the timing bound or consistently exercises both success+failure paths at all 6 sites.
- **C-CODEX-H4** — Plan 11.1-06 hardcodes "Copied — paste in terminal to **index 45 repos**" into toast copy. Repo count will drift. Plan bug, not just nit.

**MEDIUM**
- **C-CODEX-M1** — IMP-01 verification is weak (class equality + source grep). Doesn't prove rendered pixel alignment or no-clipping. GitNexus column conflict: research cites max measured 131px but plan picks `w-32` (128px) with a `w-36` escape hatch — width contract not settled.
- **C-CODEX-M2** — `--ph-h` ownership inconsistency. D-11.1-05 says "sensible scope (route container)"; Plan 04 publishes to `document.documentElement` — every PageHeader becomes a global writer. PageHeader is shared across routes → potential cross-route coupling.
- **C-CODEX-M3** — Too many implementation-proxy tests (source grep, class strings). Brittle against harmless refactors; weak against real regressions.
- **C-CODEX-M4** — Error toast copy "open the help guide for the command" doesn't match a real affordance for every clipboard action.

**LOW**
- **C-CODEX-L1** — Plans 02/03/04 all `depends_on: [01]` — only token swap actually needs to land first. Slower than necessary but not wrong.
- **C-CODEX-L2** — Verification too desktop-heavy. Donald uses iPad over Tailscale; visual checks are almost entirely 1440×900.

### Suggestions

- Replace `usePageHeaderHeight` test harness with real mounted component + callback-ref + `document.documentElement.style` assertion
- Fix Plan 06 Task 1: either mock `@tanstack/react-router`'s `Outlet`, or test the router-root layout that renders `AppShellV2`. Do not invent a `children` prop.
- Strengthen IMP-03 with one integration contract: mock `writeToClipboard` → click real CTA → assert toast present immediately + fake-timer assertion for duration bounds
- Remove hardcoded "45 repos" — either dynamic, or use stable "Copied, paste in terminal to index GitNexus"
- Tighten IMP-01: keep `<colgroup>` test, drop grep tests, add browser-level measurement / screenshot rubric, resolve `w-32` vs `w-36` BEFORE execute
- Decide `--ph-h` ownership BEFORE implementation: scope to sticky header / route container (preferred) OR accept global w/ last-writer-wins + cleanup on unmount
- Add iPad portrait QA steps to Plans 05+06 (toolbar height + top-right toast placement most at-risk on smaller viewport)
- Consider `role="alert"` for error toast (deviation from D-11.1-12) — at minimum document SR semantics test

## Plan revisions resulting from this review (PD-11.1-NN)

The following plan-time deviations are accepted and will be applied to the plans BEFORE `/gsd-execute-phase 11.1` kicks off. Recorded under CONTEXT.md `<deviations>` block.

- **PD-11.1-01 — `--ph-h` is published on `document.documentElement` (global), NOT scoped to a route container** (resolves CODEX-M2). Rationale: only one PageHeader is mounted at a time across the dashboard (single-route content model). Last-writer-wins is the intended behavior. usePageHeaderHeight RESETS `--ph-h` to the default 56px on unmount to prevent stale values from carrying across routes.
- **PD-11.1-02 — Toast copy on `IndexGitNexusButton` uses stable wording** (resolves CODEX-H4 + GEMINI-L4). Replace "Copied — paste in terminal to index 45 repos" with "Copied — paste in terminal to index your repos with GitNexus". Same calm register; no calibration-drift.
- **PD-11.1-03 — `usePageHeaderHeight` test harness uses a real mounted component** (resolves CODEX-H1). Plan 04 Task 1 test rewrite: render a `<div ref={...}>` via a test component, mock `ResizeObserver` globally in `vitest.setup.ts` (resolves CODEX-M3 + GEMINI-L2), trigger the observer callback, assert `document.documentElement.style.getPropertyValue('--ph-h')`.
- **PD-11.1-04 — Plan 06 Task 1 tests the actual mount surface** (resolves CODEX-H2). Either: (a) test the router root that wraps with AppShellV2 (`packages/spa/src/main.tsx` or the route tree definition); (b) test a `_appshell` layout component. Do NOT pass a fictional `children` prop.
- **PD-11.1-05 — IMP-03 timing assertion added** (resolves CODEX-H3). Plan 06 Task 3 must use `vi.useFakeTimers()` and assert: toast visible within 16ms of click resolution, still visible at 199ms, dismissed by 2400+200ms.
- **PD-11.1-06 — GitNexus column-width decision lock** (resolves CODEX-M1 + GEMINI-L3). Add explicit Plan 02 task between RED and GREEN-1: re-measure GitNexus column max-width against current live fixtures; if max > 124px (allowing 4px gutter), bump `w-32` → `w-36` in `coverageColumns.ts`. Record the measurement in `11.1-02-SUMMARY.md`.
- **PD-11.1-07 — `data-testid` adopted for sticky-stack queries** (resolves CODEX-M3 + GEMINI-L1). PageHeader gets `data-testid="page-header"` and `data-testid="page-header-sticky"` (when sticky=true). Plan 05 tests use `within(getByTestId(...))` instead of complex class-string selectors.
- **PD-11.1-08 — iPad portrait QA step added to Plans 05+06** (resolves CODEX-L2). Add to `<post_plan>` of each: take `/browse` screenshot at viewport 768×1024 (iPad portrait); confirm sticky stack + toast position render cleanly.

NOT adopted (rejected after review):
- **REJECT CODEX-M4** (error toast copy underspecified). Existing "open the help guide for the command" wording stands — `/help` is a single help surface and the recovery affordance is implicit. If user-test data later shows confusion, revisit in 11.2.
- **REJECT CODEX-L1** (depends_on graph too serial). The 01 → {02,03,04} → {05,06} structure is correct. Token swap genuinely must land first to avoid token-related test noise in Plans 02/03/04. The "slower than necessary" critique is the cost of safety.
- **REJECT CODEX suggestion to switch error toast to role="alert"**. D-11.1-12 stands. `role="status" + aria-live="assertive"` is the WCAG-recommended pattern for transient error messages that should announce immediately but not interrupt focus. `role="alert"` carries `aria-live="assertive"` implicitly + focus management that we don't want here.

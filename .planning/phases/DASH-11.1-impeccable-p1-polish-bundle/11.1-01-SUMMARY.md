Closes IMP-04 + IMP-05.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Plan-time WCAG math was off; chose `#706B85` over `#75708A` to satisfy stated AA-on-both-backgrounds contract**

- **Found during:** Task 2 (GREEN) — running the contrast calculator against the canonical pairs surfaced the plan-time math error.
- **Issue:** RESEARCH.md §D2 listed canonical contrast ratios that were hand-computed and off by 0.07–0.18 from the WCAG 2.1 spec output. The candidate `#75708A` chosen in D-11.1-13 was claimed to clear AA on both backgrounds (4.68 / 4.55 in the plan), but real math shows 4.52 vs app-bg (passes) and **4.38 vs sidebar-bg (fails AA 4.5 floor)**. The locked decision `#75708A` could not satisfy the locked requirement IMP-04 ("clears WCAG AA body-text contrast ≥4.5:1 against both `--color-app-bg` AND `--color-sidebar-bg`"). Behavior contract beats hex-literal contract.
- **Fix:** Selected `#706B85` instead — the closest darker candidate that genuinely clears AA on both backgrounds (4.86 / 4.71) while preserving the planned ~1.10:1 inter-tier visible-delta against secondary (`#6B6478`, 5.40 / 5.23). Test expectations in `contrast.test.ts` re-anchored to real-formula values (3.92 + 4.86) rather than plan-time approximations (3.99 + 4.69). Comment block in `tokens.css` documents the deviation with the full v1.0.0 → v1.0.1 → v1.1 history chain and a pointer back to this SUMMARY.
- **Files modified:** `packages/spa/src/lib/contrast.test.ts` (test-expectation corrections, bundled into Task 2 GREEN); `packages/spa/src/styles/tokens.css` (Task 4 GREEN).
- **Verification:** Re-computed all 6 ratios externally via Node:
  - primary `#1F1B2E` vs app-bg 16.01:1, vs sidebar-bg 15.52:1 (AAA on both)
  - secondary `#6B6478` vs app-bg 5.40:1, vs sidebar-bg 5.23:1 (AA on both)
  - tertiary `#706B85` vs app-bg 4.86:1, vs sidebar-bg 4.71:1 (AA on both)
  - 6/6 verify-contrast.test.ts assertions pass; 13/13 contrast.test.ts assertions pass; full SPA suite 913/913 pass; typecheck + build clean.
- **Committed in:** `46ef34c` (test-expectation correction, Task 2 GREEN) + `af0b4bf` (token swap, Task 4 GREEN).

**2. [Rule 3 — Blocking] Strict TS noUncheckedIndexedAccess on tuple `.map()` result**

- **Found during:** Task 2 (GREEN) — after the test suite passed, `tsc --noEmit` flagged 3 errors on `0.2126 * r + 0.7152 * g + 0.0722 * b` because `rgb.map(...)` widens `[number, number, number]` to `number[]`, losing tuple-length safety, and `r/g/b` became possibly-undefined.
- **Issue:** The plan's reference implementation in `<task type="auto" tdd="true"><behavior>` used `const [r, g, b] = rgb.map(...)`. Strict TS would not compile.
- **Fix:** Replaced the `.map()` destructure with a named `gamma(c)` helper and explicit index reads: `0.2126 * gamma(rgb[0]) + 0.7152 * gamma(rgb[1]) + 0.0722 * gamma(rgb[2])`. Same semantics; type-safe under `noUncheckedIndexedAccess`.
- **Files modified:** `packages/spa/src/lib/contrast.ts` (Task 2 GREEN).
- **Verification:** Re-ran `pnpm --filter @agenticapps/dashboard-spa typecheck` — clean. 13/13 tests still pass.
- **Committed in:** `46ef34c` (bundled into Task 2 GREEN).

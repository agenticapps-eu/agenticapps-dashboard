## INFORMATIONAL pass — 2 findings, both auto-fixed

### Issue 1 — Toast.tsx negative-duration guard (LOW)

- **Where:** `packages/spa/src/components/ui/Toast.tsx:42`
- **Problem:** `show()` accepted `duration` without clamping. A negative `duration` would call `setTimeout` with a negative value, which fires immediately in browsers — the toast would disappear before the user could read it.
- **Fix:** `const duration = Math.max(0, input.duration ?? DEFAULT_DURATION)` — 1-line clamp.
- **Action:** AUTO-FIXED in commit `381fb53`.

### Issue 2 — Toast key on portal child (LOW)

- **Where:** `packages/spa/src/components/ui/Toast.tsx:59`
- **Problem:** When `show()` is invoked while a toast is already visible (single-slot replace), React reconciles the same `<ToastView>` node and updates its `aria-live` attribute in place. Some screen readers don't re-announce on `aria-live` attribute change of an existing node — only on new node creation.
- **Fix:** Add `key={toast.id}` to the portal child so each `show()` forces a remount and fresh aria-live announcement.
- **Action:** AUTO-FIXED in commit `381fb53`. All 11 Toast.test.tsx assertions still pass (the key is opaque to the existing tests).

## Stage 2 findings (4)

### Important #1 — Toast `role`/`aria-live` contradiction (FIXED in `eea1bac`)

- **Where:** `packages/spa/src/components/ui/Toast.tsx:75-76`
- **Problem:** `role="status"` carries an implicit `aria-live="polite"` per WAI-ARIA 1.2 §5.3.3. Forcing `aria-live="assertive"` onto the same element creates conflict; the spec says explicit value wins but NVDA/VoiceOver have historically ignored the override. The rest of the codebase (`CodeBlock.tsx:46`, `ManualPairForm.tsx:245/254`) uses `role="alert"` (implicit assertive) for the assertive announcement.
- **Resolution:** Split on variant — `role={isError ? 'alert' : 'status'}`; removed explicit `aria-live` attribute. PD-11.1-12 (which originally accepted `role="status" + aria-live="assertive"`) is **reversed** by PD-11.1-10. Earlier rejection of codex's role="alert" suggestion was wrong; Stage 2 confirmed.
- **Test updates:** `Toast.test.tsx` + 4 per-site test files (`IndexGitNexusButton`, `InstallGitNexusButton`, `CoverageEmptyState`, `CoverageFamilySection`) switched error-toast assertions from `getAllByRole('status').find('Copy failed')` to `getByRole('alert')`.

### Minor #2 — Toast animation not guarded by `motion-safe:` (FIXED in `eea1bac`)

- **Where:** `Toast.tsx:77`
- **Problem:** Existing codebase (`ProjectCard.tsx:240`, `ManualPairForm.tsx:235`) honors `prefers-reduced-motion`; Toast was inconsistent.
- **Resolution:** `motion-safe:transition-opacity motion-safe:duration-150 motion-safe:ease-out`. Test `Toast.test.tsx:102` updated to assert `motion-safe:transition-opacity` instead of `transition-opacity`.

### Minor #3 — SoT drift on `<th>` actions (FIXED in `eea1bac`)

- **Where:** `CoverageFamilySection.tsx:191`
- **Problem:** Actions `<col>` correctly used `COVERAGE_COL_WIDTHS.actions`, but the corresponding `<th>` hardcoded `w-8` rather than referencing the SoT constant. Currently identical values but invariant would silently drift on future SoT edits.
- **Resolution:** `<th className={`sticky top-... ${COVERAGE_COL_WIDTHS.actions}`}>`.

<domain>
## Phase Boundary

Bundle the **four Phase-10-inherited P1s** surfaced by 11-IMPECCABLE.md into a single polish phase. Targeted lift: **Composite 76 → ~82**, Nielsen 24/40 → ~30/40. This produces **calibration data point #3** which, combined with Phase 10 (74) and Phase 11 (76), settles the D-10.5-03 floor recalibration question (provisional ≥87 floor confirmed empirically unreachable under the skill-driven gate).

**In scope (4 P1s):**

1. **Column-width lock across `CoverageFamilySection`** — each section currently renders an independently auto-widthed `<table>`; cross-family eye-tracing of a single column is broken (Repo column: 272/239/280px; GitNexus column: 92/131/100px). Fix: shared `<colgroup>` with fixed widths via `table-fixed` so all three family tables share identical column tracks.
2. **Sticky `CoverageToolbar`** — once Donald scrolls into `neuroflash`'s 33 rows the filter chips + search input scroll off; he can't see the active filter or refine search without scrolling back. Fix: fold the toolbar into the existing sticky `PageHeader` block via its `children` slot so the toolbar inherits the same sticky surface.
3. **Toast primitive wired for clipboard-write feedback** — six clipboard call sites (the "Index with GitNexus" purple CTA, family-header `Copy npm install` link, per-row `/wiki-compile` + `workflow update` popover entries, `CoverageEmptyState` install button, `InstallGitNexusButton`) silently mutate the clipboard. First-time visitors and iPad-over-Tailscale users have zero feedback. Fix: new `ui/Toast.tsx` primitive (provider + portal + `useToast()` hook) wired at every clipboard call site with success/error variants.
4. **`--color-text-tertiary` contrast lift** — current token `#807A92` renders at ~3.9-4.0:1 against `--color-app-bg` (#FAFAF7) and ~3.8:1 against `--color-sidebar-bg` (#F8F6F3); below WCAG AA body-text floor of 4.5:1. The v1.0.1 bump from `#9C95A8` (2.8:1) to `#807A92` (3.9:1) cleared the 3:1 detector floor only. Fix: swap to `#75708A` (~4.68:1 vs app-bg; ~4.55:1 vs sidebar-bg); add `verify-contrast.test.ts` invariant.

**Out of scope (P2/P3 from 11-IMPECCABLE — explicitly deferred):**

- **P2 #5** (column-header tooltips) — additive primitive (`<TooltipTrigger>`) that doesn't exist; deferred to Phase 11.2 or Phase 12 if the new IMPECCABLE re-pass shows this as the next blocker.
- **P2 #6** (family-aggregate worst-state-wins misread) — semantic redesign question; deserves a `discuss` cycle of its own. Deferred.
- **P2 #7** (`max-w-prose` on PageHeader subtitle) — one-line fix; folded INTO P1 #2's PageHeader edit only if incidental, otherwise deferred to avoid scope creep.
- **P3 #8** (collapse uniformly-missing columns when install state is `not-installed`) — IA-level behavior change; designed-against by D-11.1-09 below to preserve the calm fixed-column-grid contract Phase 11.1 establishes. Belongs in v1.2 or after the Index-GitNexus flow has been validated.

**Anti-features (must NOT happen):**

- No new third-party dependencies. Toast is hand-rolled.
- No CSS Grid refactor of `CoverageFamilySection` (rejected as Option B under D-11.1-01).
- No animation extras on Toast (no bounce-easing, no scale, no slide) — detector flags `@keyframes bounce` as a current false positive; writing our own would make it real.
- No daemon-side changes. `/cso` therefore NOT required for this phase.
- No expansion of the scope-of-text-rendered-with-text-tertiary — token swap is a single literal edit; all consumers inherit automatically.
- No mobile-specific UI rework. Existing `flex-wrap` + `table-fixed` are mobile-safe by construction.

<decisions>
## Implementation Decisions

All decisions below were brainstormed against `11-IMPECCABLE.md` recommendations and approved 2026-05-18 ("ship it" — recommended-option-across-the-board).

### Column-width lock (P1 #1)

- **D-11.1-01:** **Shared `<colgroup>` with `table-fixed`** across all three `CoverageFamilySection` `<table>` instances (NOT a single CSS-grid spanning all sections). Why: 30-LOC additive fix vs ~300-LOC refactor that would also rework the post-UAT sticky-stack layering. CSS-Grid is the right v2 architecture if multi-cell select ever lands; not the right v1.1 polish move. **Reject Option B.**
- **D-11.1-02:** Column widths defined as a single exported constant `coverageColumns.ts` (single source of truth, consumed by both `CoverageFamilySection` and the column-header sticky `<th>` slot). Widths: Repo `w-72` (288px), CLAUDE.md `w-32` (128px), GitNexus `w-32` (128px), Wiki `w-[22rem]` (352px), Workflow `w-32` (128px), Actions `w-8` (32px). Derived from current measured max-of-3 per column + ~16px stability gutter.

### Sticky toolbar composition (P1 #2)

- **D-11.1-03:** **Move `<CoverageToolbar>` from a sibling of `<PageHeader>` to a child** (use the existing `children` slot in `PageHeader.tsx`). The sticky surface is unchanged structurally; only the content inside it grows by ~50px. Reject the sibling-stick alternative ("a second sticky element at `top-14`") — multi-stick stacks compound layering risk and complicate the existing layering-stack invariant test.
- **D-11.1-04:** Separator between title row and toolbar row inside the sticky block: **gap-only (no hairline border)**. Why: matches existing `flex-col gap-1` in PageHeader; "all one sticky surface" reads better; preserves calm-instrument aesthetic.
- **D-11.1-05:** **Introduce `usePageHeaderHeight()` hook** that measures the sticky PageHeader's rendered height via `ResizeObserver` and publishes it as a CSS custom property (`--ph-h`) on a sensible scope (route container). Family-header `top-` and column-header `top-` derive from this CSS var instead of the current hardcoded `top-8` / `top-[5.0625rem]` magic numbers. Default value `56px` while first paint pending (preserves current visual). Why: avoids cascading magic-number updates across `CoverageFamilySection.tsx` every time the PageHeader's content changes; locks the layering-stack invariant in a measurable way that the test suite can assert.

### Toast primitive (P1 #3)

- **D-11.1-06:** **Top-right fixed portal, single-slot replace** semantics. Position: `top-16 right-4` (below TopBar). One toast at a time; rapid double-clicks replace the current toast. Why: TopBar already owns top chrome; toast at top-16 sits below it without collision. Bottom-center (rejected) competes with mobile virtual keyboards. Inline-per-CTA (rejected) requires every clipboard call site to manage local state — defeats the "scoped primitive" goal.
- **D-11.1-07:** **API surface (intentionally minimal):**
  ```ts
  type ToastVariant = 'success' | 'error'  // v1 — expand only on demand
  interface ToastInput { message: string; variant?: ToastVariant; duration?: number }
  interface ToastContextValue { show(input: ToastInput): void }
  const ToastProvider: React.FC<{ children: React.ReactNode }>
  const useToast: () => ToastContextValue
  ```
  Default `duration = 2400ms`. `duration: 0` = sticky until dismiss (reserved for future use, not used by Phase 11.1).
- **D-11.1-08:** **Animation:** opacity transition only (`transition-opacity duration-150 ease-out`). NO bounce, NO scale, NO slide. Why: detector currently flags `@keyframes bounce` as a Tailwind-preflight false positive; hand-rolling an animation that triggers the same pattern would make the finding real.
- **D-11.1-09:** **Provider mount point:** wrap `AppShellV2` inside the root router so every authenticated route inherits the context. Skip wrapping pre-pair routes (login / pair) — they have no clipboard writes.
- **D-11.1-10:** **Wording strategy:** contextual per call site. Clipboard-CLI-command writes → "Copied — paste in terminal to ..."; URL writes → "Copied — paste in browser". Error variant: "Copy failed — open the help guide for the command." Why: clipboard payload semantics differ; one-size-fits-all wording loses information.
- **D-11.1-11:** **Z-index:** `z-[var(--z-toast)]` (already reserved at `2000` in tokens.css:74; sits above modals at 1000 and overlays at 100 — correct for a feedback surface that must always be visible).
- **D-11.1-12:** **ARIA:** `role="status"` + `aria-live="polite"` on the toast container so screen readers announce on success without interrupting. Error variant uses `aria-live="assertive"`.

### Token swap (P1 #4)

- **D-11.1-13:** **New `--color-text-tertiary` value: `#75708A`.** Computed contrast: ~4.68:1 vs `#FAFAF7` (app-bg); ~4.55:1 vs `#F8F6F3` (sidebar-bg). Both clear WCAG AA body-text floor (4.5:1) with safe margin. Tier-gap analysis: secondary at `#6B6478` (~6.4:1) maintains a ~1.7:1 inter-tier visible delta. Rejected candidates: `#7A7388` (barely clears AA at ~4.52, monitor-drift risky), `#6B6580` (over-tightens secondary/tertiary at ~5.22:1 but only ~1.2:1 inter-tier — collapses the tier ladder).
- **D-11.1-14:** **Update `tokens.css` comment block** to document the contrast calculation, the IMPECCABLE finding date (2026-05-18), the WCAG AA citation, and the v1.0.1 → v1.1 history chain (so the next maintainer can trace why this token changed twice).
- **D-11.1-15:** **Add `verify-contrast.test.ts`** that computes actual WCAG contrast ratios for every `text-*` token against `app-bg` and `sidebar-bg` using a pure-JS WCAG 2.1 relative-luminance calculator (~10 LOC; zero deps). Asserts:
  - `text-primary` ≥ 13:1 (AAA scope)
  - `text-secondary` ≥ 4.5:1 (AA scope)
  - `text-tertiary` ≥ 4.5:1 (AA scope — the lock that closes the regression class)
  Locks the invariant: future token edits cannot regress contrast without explicit test override.

### Component naming + file boundary

- **D-11.1-16:** New files (all under `packages/spa/src/`):
  - `components/ui/Toast.tsx` — provider, hook, portal, queue (single-slot).
  - `components/ui/Toast.test.tsx` — primitive contract tests.
  - `components/ui/usePageHeaderHeight.ts` — ResizeObserver hook + CSS-var publisher.
  - `components/ui/usePageHeaderHeight.test.ts` — hook contract test (jsdom MutationObserver shim required).
  - `components/panels/coverage/coverageColumns.ts` — column-width constants (single source of truth).
  - `lib/contrast.ts` — pure-JS WCAG relative-luminance + contrast-ratio calculator.
  - `lib/contrast.test.ts` — unit tests against known WCAG canonical pairs.
  - `styles/verify-contrast.test.ts` — token-pair contrast invariant.
- **D-11.1-17:** Modified files:
  - `styles/tokens.css` — token swap + comment block update.
  - `components/ui/PageHeader.tsx` — measurement integration via `usePageHeaderHeight`.
  - `components/panels/coverage/CoveragePage.tsx` — move `CoverageToolbar` into `PageHeader children`; wrap nothing else.
  - `components/panels/coverage/CoverageFamilySection.tsx` — add `<colgroup>`; consume `coverageColumns.ts`; switch column-header `top-` from hardcoded to `var(--ph-h)`.
  - `components/panels/coverage/CoverageRow.tsx` — same column-track adoption inside `<tbody>` rows.
  - `components/AppShellV2.tsx` (or router root) — wrap with `<ToastProvider>`.
  - All 6 clipboard call sites — replace `await writeToClipboard(x)` bare invocations with `await writeToClipboard(x) && toast.show({ message: '...' })`.

The data, not opinion, decides.

### Token + invariant locks

- **D-11.1-22:** The shared `coverageColumns.ts` widths become **invariant-locked** — `coverageColumns.test.ts` asserts the constant is consumed by both `CoverageFamilySection` and `CoverageRow` and that no per-column literal widths drift in.
- **D-11.1-23:** `usePageHeaderHeight` becomes the **single source of truth** for the layering-stack `top-` offsets. The two existing magic numbers (`top-8`, `top-[5.0625rem]`) are deleted, not preserved. Forward compat: any future child added to `PageHeader children` automatically slides the offsets without touching `CoverageFamilySection`.

### Claude's Discretion

- **Exact pixel widths** in `coverageColumns.ts` — planner can re-measure on the running dev server and adjust ±8px from the values in D-11.1-02 if a measurement deltas shows it. Constraint: widths MUST be fixed (no `auto`).
- **Toast portal target** — `document.body` vs a dedicated `#toast-root` div. Planner picks based on test ergonomics; the IMPECCABLE-affected surface is unchanged either way.
- **Whether `lib/contrast.ts` accepts hex strings or RGB tuples** — pure-function shape; planner picks the API that tests cleanest.
- **ResizeObserver fallback** for browsers lacking the API (none in practice — supported everywhere SPA already runs). Planner can add a `typeof ResizeObserver === 'undefined'` early return that keeps `--ph-h` at the 56px default.

Phase 11.1 mints 5 new requirements. All map directly to a P1 fix from 11-IMPECCABLE.md.

<risks>
## Risk register

- **R1** — `usePageHeaderHeight` ResizeObserver may flicker on first paint before measurement lands. Mitigation: `--ph-h: 56px` default in `:root`; ResizeObserver replaces it on first measurement (~16ms).
- **R2** — Combined sticky title+toolbar exceeds `40vh` on iPad portrait (rare — title is ~40px, toolbar is ~50px, total ~90px ≪ 40% of 1024px). Mitigation: not needed; document as a non-issue.
- **R3** — Token swap affects every site rendering `text-text-tertiary` (~40 occurrences across `packages/spa/src/`). Some chrome-text may visually shift "darker". Mitigation: design pass during execution; the IMPECCABLE re-pass at gate validates no regression at 1440×900 across all v1.1 routes.
- **R4** — Toast portal `document.body` mount races against React StrictMode double-render in tests. Mitigation: idempotent root creation in `Toast.tsx` (check existing `#toast-root` before insert).
- **R5** — `<colgroup>` widths fixed in px; very narrow viewports (~600px) may horizontal-scroll inside the family section. Mitigation: matches existing behavior (table already overflows below ~720px). Documented as a Phase 12 candidate ("Coverage responsive collapse").

</risks>

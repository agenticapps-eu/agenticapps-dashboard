## Background

Two surfaces share the name "impeccable" and were conflated by Phase 06:

| Surface | Source | Status |
|---|---|---|
| The npm CLI `npx impeccable` | npm package | `critique` subcommand was removed in v1.0.1 and v1.x was unpublished from npm. Only `impeccable detect` exists in v2.x. Output shape is a finding list, not a `{routes: [{score}]}` scalar. |
| The Claude skill `impeccable:critique` | `~/.agents/skills/impeccable/SKILL.md` (vendored) | Alive. `critique [target]` subcommand documented in `command-metadata.json` — described as "quantitative scoring, persona-based testing, automated anti-pattern detection, and actionable feedback." LLM-driven: spawns sub-agents to read source + visit live URLs, returns scored heuristic review. |

The dashboard's Phase 6 gate (`.github/workflows/impeccable.yml`, `scripts/check-impeccable-score.mjs`, `scripts/screenshot.mjs`) was built on the CLI when its `critique` subcommand still existed. Since the rename, the workflow fails at the `npx impeccable critique` step. Phase 10 Gate 4 is blocked on the same drift.

The AgenticApps workflow's planning hooks (`claude-workflow/templates/config-hooks.json:27`, `agenticapps-workflow-core/adrs/0011-...md`) reference `impeccable:critique` as a **skill** hook. Those are not broken — they invoke the LLM skill, which still has the subcommand.

## Goal

Replace the broken CLI gate with a skill-driven per-phase artifact. No CI replacement.

**Each frontend-touching phase ends by running `/impeccable critique` and committing `<phase>-IMPECCABLE.md` alongside other phase docs (`*-REVIEW.md`, `*-QA.md`, `*-VERIFICATION.md`).** The artifact records:
- Composite score + sub-scores
- Findings + remediation recommendations
- Verdict against the ≥ 87 floor (Phase 06.1 D-6-09.v1 amendment, carried forward)

PR-time enforcement is a discipline issue, not a CI gate. The skill's coverage (description: "automated anti-pattern detection" plus hierarchy/cognitive-load/emotional-resonance review) is a strict superset of what the CLI did. There's no need for a separate deterministic CI check.

## First-principles justification for dropping CI automation

Earlier draft of this scope proposed a "two-layer gate" (CI smoke via `impeccable detect` + per-phase skill artifact). Dropped after pushback that asked: what would the CI layer catch that the skill misses?

- **Coverage**: skill includes anti-pattern detection per its own SKILL.md.
- **Regression-between-phases**: the existing CI gate has been silently broken for weeks and nothing regressed. The risk is theoretical, not measured.
- **External-contributor enforcement**: this is a private dashboard, no external contributors.
- **`/review` already runs against every diff** and flags obvious anti-patterns at PR time.
- **Per-phase artifacts + git history** are a stronger signal than green-check CI for this codebase.

The skill being LLM-driven and run-to-run non-deterministic is fine for a human-reviewed phase artifact. Acceptable tradeoff.

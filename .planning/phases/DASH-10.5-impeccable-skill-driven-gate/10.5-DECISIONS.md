# Phase 10.5 — Decisions

Decisions adopted as part of Phase 10.5 (impeccable skill-driven gate). Follow the dashboard's existing `D-N-NN` decision-naming convention rather than introducing a standalone ADR directory.

These decisions supersede the parts of Phase 6's D-6-09 / D-6-09.v1 / D-6-10 / D-6-11 / D-6-21 that specified CI enforcement of the impeccable gate. Their score-floor and breakpoint commitments remain in force, applied to the new per-phase artifact.

## D-10.5-01 — Retire the CI-enforced impeccable gate

The `.github/workflows/impeccable.yml` workflow, `scripts/check-impeccable-score.mjs`, and its tests are deleted. Phase 6's CI gate has been silently broken since `npx impeccable critique` was removed (v1.0.1, v1.x unpublished from npm). No replacement CI workflow is created.

**Why:**
- The CLI rename is upstream and not reversible — v1.x is gone from npm; only `impeccable detect` exists in v2.x, with a different (finding-list) JSON shape that doesn't produce a 0-100 composite. Adapting the score script to `detect` would be a maintenance burden without measured benefit.
- The CI gate has been broken for weeks with no measured regression — clear evidence it wasn't load-bearing for actual quality.
- The vendored Claude skill `~/.agents/skills/impeccable/SKILL.md` still has `critique` and is strictly more powerful (LLM + persona checks + automated anti-pattern detection in the same pass). Two layers (CI + skill) would have overlapping coverage.

**How to apply:** Delete the three files. Update `CLAUDE.md:38` + `docs/spec/dashboard-prompt.md` lines 554, 594, 696 to describe the per-phase artifact contract. Reference this decision file from each updated line.

## D-10.5-02 — Skill-driven per-phase artifact is the gate

Every frontend-touching phase ends by running `/impeccable critique` against the affected routes at 1440×900 and committing `<N>-IMPECCABLE.md` alongside the phase's other artifacts (`*-REVIEW.md`, `*-QA.md`, `*-VERIFICATION.md`).

The artifact carries:
- Composite score + per-heuristic (Nielsen × 10, scored 0–4 each) + sub-scores
- Cognitive-load checklist results
- Persona red flags (2–3 personas relevant to the surface)
- Priority issues with P0–P3 severity + suggested-command tagging
- AI-slop verdict (both LLM judgment and deterministic detector output)

**Why:**
- The skill is LLM-driven and run-to-run non-deterministic — but the artifact is human-reviewed and committed, so the variability is captured at a single point and judged in context.
- Two-assessment isolation (LLM in one sub-agent, deterministic detector in another, no shared output) is mandated by the skill's `critique.md` reference and preserved by running each in a separate `Agent` invocation.
- Per-phase placement means the artifact lives in commit history next to the code that produced it — better signal than a CI green-check at PR time.

**How to apply:** Add `/impeccable critique` as the canonical Gate 4 step in every phase's HUMAN-UAT / closure checklist. When `/gsd-execute-phase` post-hooks gain skill-invocation support upstream (claude-workflow concern), wire it there.

## D-10.5-03 — Composite floor ≥ 87, calibration pending

Carry forward Phase 06.1's D-6-09.v1 amendment — composite ≥ 87 at 1440×900 — as the floor for the new per-phase artifact. Mark this as **provisional** until the skill-driven scoring has produced three phase artifacts; recalibrate then if the skill's natural mid-band differs meaningfully from the old CLI's mid-band.

**Why:**
- The 87 floor was set against the old CLI's scoring distribution. There's no a-priori reason it maps cleanly to the skill's Nielsen-heuristic × 10 + AI-slop synthesis.
- Phase 10's first artifact landed at ~74 — well below 87. Two interpretations: the page has real P0 issues (which the artifact identifies), or the skill scores more strictly than the old CLI. Both may be true.
- Locking the floor permanently before three calibration data points would be premature.

**How to apply:** After three frontend phases produce `<N>-IMPECCABLE.md` artifacts under this regime, write `D-10.5-03.calibration` recording the observed band and either confirming 87 or recalibrating. Until then, individual phases below 87 should fix-and-rerun before merging, with a written waiver in their VERIFICATION.md if the floor is genuinely unreachable for a phase-specific structural reason (mirroring D-6-09.v1's `/projects/:id` precedent).

## D-10.5-04 — No headless skill invocation in CI

The skill MUST be invoked by a developer in their Claude Code session. No `claude --headless` invocation in CI to automate the per-phase artifact.

**Why:**
- LLM cost per run is non-trivial (sub-agents + browse + source reads).
- Skill is run-to-run non-deterministic; CI flakiness amplifies the cost.
- The artifact is a human-reviewed artefact, not a green-check. Making it CI-enforced removes the review step's value.
- A skipped or absent artifact is itself a visible signal in PR review.

**How to apply:** Phase closure checklist enforcement only. If artifact absence becomes a recurring issue, add a `pre-merge` hook (locally, not CI) that warns when `<N>-IMPECCABLE.md` is missing for a phase that touched `packages/spa/`.

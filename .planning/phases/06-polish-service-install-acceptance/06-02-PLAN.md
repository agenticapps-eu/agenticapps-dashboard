<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated SPA -> daemon mutation routes | Bearer-token holder can spam mutation routes; rate-limit caps damage budget per minute |
| SPA-supplied name/tags -> registry.json on disk | Unbounded lengths could write multi-MB strings to a 0600 file (DoS via storage exhaustion) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-02-01 | Denial of Service | /:id/rename, /:id/tags, /register-confirm | mitigate (A-01) | Sliding 10s window + 10-burst rate limit per token-hash, mirroring the existing register-prepare gate at registry.ts:137-150 |
| T-06-02-02 | Denial of Service | RenameRequestSchema.name, TagsRequestSchema.tags | mitigate (A-02) | `.max(200)` on name; `.max(20)` on tag array; `.max(50)` per tag — caps registry.json bloat to ~5KB/project worst case |
| T-06-02-03 | Tampering | rate-limit token-hash key | accept | Token holder with multiple SPA tabs is one logical actor; the per-token bucket is the right granularity. A token-rotation between requests would issue a fresh tokHash and reset the bucket — acceptable since token rotation is an explicit user action with its own threat model |
| T-06-02-04 | Information Disclosure | requestId echoed in 429 body | accept | requestId is a v4 UUID with no token material; same disclosure surface as existing 4xx responses |
</threat_model>

## Tooltip primitive (P1 #1 — column-header explanations)

| Option | Description | Selected |
|--------|-------------|----------|
| Native `title=` attribute | Zero deps, zero code. No keyboard-focus reveal, mobile invisible, screen-reader inconsistent. ~60% fix. | |
| `<abbr title=>` | Semantic, native dotted underline. Same a11y limitations as raw `title=`. | |
| `@radix-ui/react-tooltip` | Battle-tested, full a11y. Adds ~10KB gzip dep. Violates D-5.1-10 "no shadcn aliases" precedent and Phase 11.1's "build small primitives in-house" pattern. | |
| **Local `Tooltip.tsx` primitive (~60–80 LOC)** | **Hover + focus reveal, `aria-describedby`, dotted-underline trigger. Matches Phase 11.1 Toast architecture. Zero new deps.** | ✓ |

**Resolution rationale:** D-5.1-10 explicitly forbids shadcn aliases (implicitly Radix); Phase 11.1 set precedent that "small in-house primitives" is the project's design language (Toast.tsx). Native `title=` is a ~60% fix and the persona red flag in `11.1-IMPECCABLE.md` ("first-time visitors don't know what GitNexus means") merits the proper ~95% fix.

**Follow-up sub-decisions resolved:**

- **API shape:** `<Tooltip content="...">{children}</Tooltip>` — wraps trigger, no provider needed (per-instance state).
- **Reveal behavior:** mouseenter + focus open, mouseleave + blur + Escape close. 100ms open delay (avoid flicker on quick mouse pass), 0ms close (snappy).
- **Animation:** opacity-only, 100ms in / 0ms out — honors Phase 11.1 D-11.1-08 motion floor.
- **Trigger style:** subtle `border-b border-dotted border-text-tertiary` — no `?` glyph noise on the header.
- **Z-index:** `var(--z-overlay)` = 100 (above sticky 10, below modal 1000).
- **Content strings:** explicit per-column wording (CONTEXT D-11.2-05), housed in sibling `coverageColumnTooltips.ts` SoT alongside `coverageColumns.ts` for future i18n.

## Per-row in-flight feedback for gitnexus-analyze (P1 #2)

| Option | Description | Selected |
|--------|-------------|----------|
| Per-row `useMutation` instance | One mutation handle per row; isolated pending state. Multiplies mutation surface and the daemon serialises anyway. | |
| Local optimistic state in `CoverageRow` | Set local flag on click, clear on settle. Drifts from server truth on error; loses auto-invalidate refetch. | |
| **Fan-out `refresh.isPending` + `.variables` from `CoveragePage` → `CoverageRow` via derived `pending` prop** | **Single mutation handle; per-row truth from TanStack v5's exposed `variables`. No new state plumbing.** | ✓ |

**Resolution rationale:** `useCoverageRefresh` already returns `UseMutationResult` with `.isPending` + `.variables`. The daemon serializes gitnexus-analyze (`RefreshAllStaleButton.tsx:52` comment confirms). Single-mutation pattern + fan-out derived prop is the conventional TanStack v5 approach.

**Visible signal sub-decisions:**

- Refresh icon swaps `<RefreshCw>` → `<RefreshCw className="animate-spin" />` (Tailwind built-in keyframe).
- `aria-busy="true"` on both the button AND the `<tr>` for screen reader announcement.
- Button receives `disabled` to prevent popover-open mid-flight.
- Refresh button forced `opacity-100` while pending — overrides `group-hover` so spinner stays visible if pointer leaves row.
- Success toast added: `Indexed <family>/<repo>` via existing `useToast()`. Error toast `role="alert"` (Phase 11.1 PD-11.1-10 wiring).

## Wiki column width (P2 #3)

| Option | Description | Selected |
|--------|-------------|----------|
| Keep `w-[22rem]` (352px), add `truncate` + `title=` | Conservative; preserves max-content rendering for the rare long sha+timestamp case. Doesn't address whitespace concern. | |
| `w-60` (240px) with `truncate` | The recommendation in `11.1-IMPECCABLE.md`. Aggressive — 240px tight against `<sha> · <relative-time>` worst-case (~150px) + breathing space. Risk of truncation on edge cases. | |
| **`w-72` (288px), no truncate needed** | **Matches Repo column width (`w-72`) for visual rhythm. 288px - 150px = 138px breathing room. Cleaner than `w-60`+truncate, less wasteful than `w-[22rem]`.** | ✓ |

**Resolution rationale:** `w-60` + truncate is two changes; `w-72` is one and produces a visually balanced left side of the table (Repo + Wiki both 288px). The whitespace concern in IMPECCABLE was about *352px* feeling empty; *288px* lands inside comfortable. Plan-time measurement will confirm against actual `<CoverageCell column="wiki">` rendering.

## Refresh-icon touch target (P2 #4)

| Option | Description | Selected |
|--------|-------------|----------|
| Increase icon size from 14px → 24px | Visually heavier; breaks current density. Still doesn't reach 44px button without padding. | |
| Wrap icon in `min-w-[44px] min-h-[44px]` button + bump padding | Icon visual stays small; click area meets HIG. Actions column needs to widen. | ✓ |
| Replace icon-only with icon + label button | Layout change, more invasive. Phase 11.2 scope is polish, not layout overhaul. | |

**Resolution rationale:** Apple HIG 44×44pt. Padding-expansion preserves Phase 11.1's clean visual density (small icon, large hit area). Actions column widens `w-8` (32px) → `w-12` (48px). Net table-width change: −48px (the Wiki tightening at D-11.2-09 frees 64px; +16px on actions; net narrower).

**Focus ring caveat:** The `focus-visible:ring-2` stays drawn against the icon, not the 44px hit area — preserves Phase 11.1's tight focus visual.

## Controlled search input (P3 #5)

| Option | Description | Selected |
|--------|-------------|----------|
| `key={search}` to force remount on URL change | One-line. Causes flicker on URL state-write echo loop. | |
| Lift value to parent state | Cleanest controlled pattern. Loses local debounce isolation that Phase 10 deliberately introduced. | |
| **Mirror state + `useEffect` sync on prop change; debounce stays in toolbar** | **Preserves debounce locality; URL back-button changes re-seed the input via the effect.** | ✓ |

**Resolution rationale:** Phase 10 made an explicit choice to keep debounce inside the toolbar for testability + isolation. Lifting value upstream would unwind that. Mirror-state pattern is the conventional React answer.

**Sketch:**

```tsx
const [inputValue, setInputValue] = useState(search)
useEffect(() => { setInputValue(search) }, [search])
function handleSearchChange(e: React.ChangeEvent<HTMLInputElement>) {
  setInputValue(e.target.value)
  if (debounceRef.current) clearTimeout(debounceRef.current)
  debounceRef.current = setTimeout(() => onSearchChange(e.target.value), 200)
}
// <input value={inputValue} ... />
```

## Implementation Decisions

### Tooltip primitive (P1 #1)

- **D-11.2-01:** **Build a local `Tooltip.tsx` primitive** in `packages/spa/src/components/ui/Tooltip.tsx` (~60–80 LOC). No new dependencies. Rejected alternatives:
  - Native `title=` attribute: no keyboard-focus reveal, mobile invisible, screen-reader inconsistent. ~60% fix, leaves persona red flag partially open.
  - `<abbr title=>`: same accessibility limitations as raw `title=`; semantic but UX-inadequate.
  - `@radix-ui/react-tooltip`: violates project's "in-house small primitives" precedent set by Phase 11.1's Toast primitive. Adds ~10KB gzip dep. The dashboard's `D-5.1-10` constraint comment in `CoverageRow.tsx` explicitly forbids "shadcn aliases" — implicitly discourages Radix UI adoption.
- **D-11.2-02:** **API shape:** `<Tooltip content="...">{children}</Tooltip>` wraps the trigger. Trigger receives a subtle `border-b border-dotted border-text-tertiary` style + `tabindex="0"` + `aria-describedby={tooltipId}`. Floating element is `position: absolute`, top-full + small offset, z-index `var(--z-overlay)` = 100 (above sticky `--z-sticky` = 10, below modal `--z-modal` = 1000).
- **D-11.2-03:** **Reveal behaviour:** show on `mouseenter` + `focus`, hide on `mouseleave` + `blur` + Escape key. No long-press logic — `tabindex="0"` + on-screen-keyboard focus is the iPad-via-VoiceOver path. 100ms open delay, 0ms close (snappy hide; matches Phase 11.1 Toast precedent of opacity-only no-motion).
- **D-11.2-04:** **Animation:** opacity-only fade-in (no slide/scale/bounce), 100ms in / 0ms out. Matches Phase 11.1 D-11.1-08 motion floor (`reduced-motion` already covered by opacity-only).
- **D-11.2-05:** **Column-header tooltip content** (deliberate, terse, first-time-reader friendly):
  - `CLAUDE.md`: "Project AI instructions file. Must exist in repo root for AI coding agents to pick up project conventions."
  - `GitNexus`: "Local code index for repo-aware AI search. Built by `gitnexus analyze`; stored under `~/.gitnexus`."
  - `Wiki`: "Compiled knowledge base from CLAUDE.md, ADRs, READMEs. Built by `/wiki-compile`."
  - `Workflow`: "Installed version of `agenticapps-workflow`. Compared against the current scaffolder release."
  (These strings live in `coverageColumnTooltips.ts` next to `coverageColumns.ts` as a sibling SoT — same locality, single string-table for future i18n.)

### Per-row in-flight feedback (P1 #2)

- **D-11.2-06:** **Wire `useCoverageRefresh.isPending` + `.variables` down to `CoverageRow`** via a new derived `pending` prop. The mutation lives in `CoveragePage.tsx:85`; TanStack Query v5 exposes `refresh.variables` while pending. Compute `pending = refresh.isPending && refresh.variables?.action === 'gitnexus-analyze' && refresh.variables.family === row.family && refresh.variables.repo === row.repo` in `CoverageFamilySection` and pass `pending` through to each `CoverageRow`. Rejected alternatives:
  - Per-row `useMutation` instance: would multiply mutation surface; daemon serialises gitnexus-analyze anyway (`RefreshAllStaleButton.tsx:52`), so the single-mutation pattern is correct.
  - Local optimistic state in `CoverageRow`: drifts from server truth on error; loses the auto-invalidate refetch already wired in `onSuccess` (`coverageQueries.ts:114`).
- **D-11.2-07:** **Visible signal:** when `pending === true` for a row:
  - Refresh icon swaps from `<RefreshCw>` to `<RefreshCw className="animate-spin" />` (Tailwind built-in).
  - Refresh button receives `aria-busy="true"` + `disabled` (prevents popover open mid-flight).
  - Row `<tr>` receives `aria-busy="true"` for screen readers.
  - Refresh button visibility is forced to `opacity-100` regardless of `group-hover` state (so the spinner stays visible even when the user moves the pointer off-row).
- **D-11.2-08:** **Toast on settle:** existing `onSuccess` already invalidates queries. Add a success toast `Indexed <family>/<repo>` via the existing `useToast()` provider (~3 LOC in `CoveragePage.tsx:139` `case 'gitnexus-analyze'`). Error path posts an error toast `Indexing failed: <reason>` with `role="alert"` (the Phase 11.1 PD-11.1-10 wiring stands).

### Wiki column width (P2 #3)

- **D-11.2-09:** **Tighten `wiki` from `w-[22rem]` (352px) to `w-72` (288px).** Rejected `w-60` (240px) — the recommendation in `11.1-IMPECCABLE.md` — as too aggressive; 288px matches the repo column width (`w-72`) for visual rhythm and leaves comfort room for the worst-case `<sha> · <relative-time>` rendering (`f3a7b21 · 6d ago` measures ~150px; `never compiled` measures ~110px). No `truncate` utility needed at 288px — content fits. Adds 64px of horizontal breathing room at the table level, plus 64px headroom below 768px viewport breakpoint.
- **D-11.2-10:** **`coverageColumns.ts` comment chain updated** — preserve the history-of-changes pattern Phase 11.1 introduced for `--color-text-tertiary` (in `tokens.css`). Add `// PD-11.2-XX: 22rem → w-72 (288px) — 11.1-IMPECCABLE.md P2 #3, w-[22rem] over-allocated for max measured 150px content`.

### Touch target (P2 #4)

- **D-11.2-11:** **Refresh button hit area** — wrap the icon button with `min-w-[44px] min-h-[44px]` (Apple HIG). The 14px `RefreshCw` icon stays small (visual density preserved); the button's padding expands to `p-[15px]` so `2*15 + 14 = 44`. The visible focus ring (`focus-visible:ring-2`) maintains its current geometry against the icon, not the hit area — preserves Phase 11.1's clean focus visual.
- **D-11.2-12:** **Actions column width:** bump `actions: 'w-8'` (32px) to `actions: 'w-12'` (48px) in `coverageColumns.ts`. 48px = 44px button + 4px breathing room. The 16px increase is absorbed by the wiki column tightening (D-11.2-09 frees 64px) — net horizontal change is **−48px** (table still narrower).

### Controlled search input (P3 #5)

- **D-11.2-13:** **Hybrid controlled-with-prop-sync pattern.** Keep the 200ms debounce inside `CoverageToolbar` (encapsulation); add internal mirror state synced from prop on change:
  ```tsx
  const [inputValue, setInputValue] = useState(search)
  useEffect(() => { setInputValue(search) }, [search])
  // handleSearchChange: setInputValue(e.target.value); debounce(onSearchChange, 200)
  // <input value={inputValue} ... />
  ```
  Rejected `key={search}` remount — flickers on URL state-write echo loop. Rejected lifting value entirely to parent — loses the local debounce isolation Phase 10 deliberately introduced. The mirror-state pattern preserves debounce locality while making URL back-button changes re-seed the input.

### PageHeader subtitle line length (P3 #6)

- **D-11.2-14:** **One-line edit** in `PageHeader.tsx:58`. Add `max-w-prose` to the helper paragraph's className: `<p className="mt-1 text-sm text-text-tertiary max-w-prose">{helper}</p>`. `max-w-prose` = 65ch — squarely inside the WCAG-recommended 45–75ch reading band. No prop API change; applies retroactively to all 4 routes using `PageHeader` (Coverage, Skills, Pair, Home).

### Regression contract (Phase 11.1 must-haves stay intact)

- **D-11.2-15:** Phase 11.1's 5 IMP-01..IMP-05 must-haves and 4 Phase 11 must-haves form the regression contract. Plans MUST include verification that:
  - **IMP-01** column-width lock still works (after Wiki tightening + actions widening): scroll-400 shows all 3 family sections share identical column edges using the new widths.
  - **IMP-02** sticky-toolbar `--ph-h` math holds (the only PageHeader change is `max-w-prose` on subtitle — no height change expected, but `usePageHeaderHeight` ResizeObserver should still publish a stable `--ph-h`).
  - **IMP-03** Toast wiring unchanged at 6 sites; D-11.2-08 adds a 7th (gitnexus-analyze success).
  - **IMP-04** `--color-text-tertiary: #706B85` unchanged. (Tooltip arrow/border colors must NOT introduce new hex literals — use existing `border-divider-soft`/`bg-card-bg`.)
  - **IMP-05** `verify-contrast.test.ts` invariant still passes; new tooltip text-on-bg pair (tooltip body uses `text-text-primary` on `bg-card-bg` — already covered by existing assertions).

### Claude's Discretion

- **Tooltip arrow rendering:** small SVG triangle vs CSS `border` trick vs no arrow at all. Decided by plan author — both work; arrow improves discoverability. Plan should pick one and TDD-cover the visual.
- **Tooltip auto-positioning:** if the trigger is near the right edge of the viewport, the tooltip could overflow. For column headers (which are mid-table), this isn't an issue. Plan can defer collision-detection logic.
- **Test ergonomics for `pending` prop:** the existing `CoverageRow.test.tsx` has well-established patterns for prop-driven rendering. Adding a `pending` test case follows the established style.

### Folded Todos

<specifics>
## Specific Ideas

- **"Subtle, not ostentatious"** is the tooltip visual language: dotted underline on the trigger (1px, `text-text-tertiary` dotted) — no `?` glyph noise on the header, no help-icon clutter. Goal: discovery without visual debt.
- **"Spinner stays visible when pointer leaves row"** for in-flight feedback. The `group-hover:opacity-100` pattern breaks during a slow scan if the user moves their mouse. Force `opacity-100` while `pending === true` so the row's status remains legible.
- **PR #36's discipline note** (Phase 11.1 handoff): "Future docs-only STATE.md / session-handoff updates need their own small PR." Phase 11.2 honors this — feature branch + PR for the implementation, then a separate small PR for post-merge STATE/ROADMAP updates.

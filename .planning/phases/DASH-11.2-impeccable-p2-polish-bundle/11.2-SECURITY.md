## Trust Boundaries

Phase 11.2 crosses **no trust boundaries**. The hard architectural constraints
documented in `CLAUDE.md` (read-only project filesystems, path allow-list per
project, daemon writes confined to `~/.agenticapps/dashboard/`, bearer-token
auth on every route, CORS locked to `https://dashboard.agenticapps.eu` /
`http://localhost:5174`) are unchanged. The only daemon call exercised by this
phase — `POST /api/coverage/refresh` for `gitnexus-analyze` — was introduced in
Phase 10 and validated by the existing `CoverageRefreshRequest` Zod schema.

| Boundary | Description | Data Crossing in 11.2 |

| Browser ↔ Local daemon | SPA fetch to `127.0.0.1:5193` with bearer token | None new — `POST /api/coverage/refresh` already exists from Phase 10; per-row trigger surface is a UI wiring change, not a new boundary |
| SPA bundle internals | React component tree | All 11.2 changes are intra-bundle React renders; no IPC, no postMessage, no dynamic-code-eval, no raw-HTML injection sinks |

## Threat Register

All eight threats inherit the SPA's standing defenses: React JSX auto-escaping
of text nodes, TypeScript-enforced `string` prop types, and the absence of
any raw-HTML injection sink (no innerHTML write, no React unsafe-HTML escape
hatch) anywhere in the modified files. None of the threats requires active
code mitigation beyond the disposition recorded in the source plan.

| Threat ID | Category | Component | Disposition | Mitigation | Status |
|-----------|----------|-----------|-------------|------------|--------|
| T-11.2-01 | Tampering (XSS via tooltip content) | `Tooltip.tsx` `{content}` rendering | accept | `content: string` is TS-enforced; rendered via React JSX text node (auto-escaped); no raw-HTML sink. Production call sites in `CoverageFamilySection.tsx:194-197` pass hardcoded constants from `coverageColumnTooltips.ts`. Portal fix (commit `e4260b0`) preserves the same JSX text-node render path inside `createPortal(panel, document.body)`. | closed |

| T-11.2-03 | Information Disclosure (daemon error leaks paths) | `CoveragePage` error toast `Indexing failed: ${err.message}` | accept | Daemon stderr is curated by the existing `CoverageRefreshResponse` error path (Phase 10 INV-04 schema validation). `gitnexus` binary errors are short technical messages (`"exit code 1"`, `"spawn ENOENT"`) — no absolute paths or tokens. ### Post-execution remediation review

Two commits landed after the original 11.2-{N}-SUMMARY files were written:

- **`e4260b0` — Tooltip portal fix.** Swaps the panel render path from
  `<span class="absolute ...">` to `createPortal(panel, document.body)` with
  `position: fixed` + viewport coords from `getBoundingClientRect()`. Adds
  `window.addEventListener('scroll', ..., true)` and `'resize'` listeners
  while the panel is open. **No new attack surface:** the `content` prop type
  is unchanged (`string`, TS-enforced), the render path is still a React JSX
  text node, no DOM HTML APIs are used, no postMessage / cross-origin
  communication, no dynamic-code-eval. T-11.2-01 / T-11.2-02 dispositions
  unchanged.
- **`34eca52` — CoverageToolbar debounce cleanup (WR-01).** Adds an empty-deps
  `useEffect` cleanup that clears the existing `debounceRef` on unmount. Pure
  defensive timer hygiene. T-11.2-07 disposition unchanged.

### Priority Issues (new — for follow-up cycles)

#### [P1] Tooltip panel width constrained by `table-fixed` containing block — NEW, introduced by 11.2-01

**Where:** `packages/spa/src/components/ui/Tooltip.tsx:55-58`. The `<span class="relative inline-block">` trigger wrapper sits inside a `<th>` whose containing block is sized by `table-fixed` to the column's allocated width (CLAUDE.md = 128px `w-32`; GitNexus = 144px `w-36`; Wiki = 288px `w-72`; Workflow = 144px `w-36`). The absolute-positioned panel's max-width is therefore the cell minus padding — `max-w-xs` (320px) is overridden.

**Measured behavior at 1440×900:** CLAUDE.md tooltip renders at `clientWidth: 98px` (`clientHeight: 187px`), wrapping 105 characters into ~11 lines. GitNexus and Workflow tooltips ≤106px wide. Wiki (288px column) renders correctly because the column itself is wide enough.

**Why it matters:** The tooltip primitive closes a 4-phase P1 debt (H6) but its rendering degrades the help UX. A narrow-tall tooltip for a first-time visitor trying to understand column vocabulary undermines the intended affordance — the copy is excellent, but the rendering reads as "broken or truncated." This is the only regression introduced by Phase 11.2's own work.

**Fix:** Render the tooltip panel via `createPortal(panel, document.body)` rather than as a sibling `<span>`. Portal escapes all containing blocks; positioning via the trigger's `getBoundingClientRect()`. Same pattern as `Toast.tsx`. Estimated ~20 LOC + 1–2 new tests in `Tooltip.test.tsx`. Alternative quick fix: `whitespace-normal min-w-[240px]` + `left-1/2 -translate-x-1/2` centering, but verify no clipping at far-left/right table edges.

**Suggested command:** `$impeccable harden` (portal escape, recommended) — or fold into a single follow-up commit on this branch before opening the PR. Estimated lift: +2 composite (clears ≥85).

#### [P2] Tooltip dotted-underline discovery cue may be sub-pixel at `text-xs` column header scale

**Where:** `Tooltip.tsx:62` — `border-b border-dotted border-text-tertiary cursor-default`. The dotted underline is implemented via `border-bottom`, not `text-decoration` (correct per D-5.1-10's "no shadcn / no CSS hack" rule). However, at `text-xs text-text-tertiary` column header scale (~10px rendered), the dotted-border may be visually sub-pixel or blend against the column header's own text color.

**Why it matters:** If the underline doesn't read as a discovery cue, mouse users won't hover. Keyboard users discover via Tab regardless.

**Fix:** Either (a) increase underline contrast (`border-text-secondary` instead of `border-text-tertiary`), or (b) switch trigger cursor from `cursor-default` to `cursor-help` as an explicit indicator.

**Suggested command:** `$impeccable colorize`

#### [P2] Wiki column at 288px still wider than median content in neuroflash family

**Where:** `coverageColumns.ts:10` `wiki: 'w-72'`. The tightening from 352→288 matches the Repo column. But neuroflash's 33 repos mostly show "never compiled" or "repo not in .wiki-compiler.json sources" (≤200px). At 288px there's 80–168px right-padding per cell. The 11.2-CONTEXT.md explicitly chose 288 over 240 to accommodate the "sha + timestamp" worst-case.

**Why it matters:** Low-priority aesthetic density rather than usability. Total table width = 1137px on 1440px viewport — no horizontal scroll.

**Fix:** Consider `w-60` (240px) with `truncate` + `title=` fallback, OR accept the tradeoff. Not blocking.

**Suggested command:** `$impeccable layout` (low priority)

#### [P3] Tooltip panel has no directional arrow

**Where:** `Tooltip.tsx:71` — the panel renders as a floating card with no `::before` triangle connecting to trigger. The 11.2-CONTEXT.md noted "small SVG triangle vs CSS border trick vs no arrow at all" as a discretionary call; chose none.

**Why it matters:** Combined with the narrow-panel issue (P1), the tooltip reads as an absolutely-positioned help paragraph rather than a tooltip primitive.

**Fix:** 6px CSS triangle via `::before` pseudo-element on the panel. Best done after the portal fix (P1) so the arrow positioning is calculable.

**Suggested command:** `$impeccable shape`

#### [P3] No "Repo" column header tooltip — visual asymmetry

**Where:** `CoverageFamilySection.tsx` — CLAUDE.md, GitNexus, Wiki, Workflow `<th>` elements have `<Tooltip>` wrappers; Repo `<th>` does not. "Repo" is self-explanatory but the asymmetry (4 dotted underlines, 1 plain) may create a "did this column lose its tooltip?" moment.

**Why it matters:** Trivial visual consistency. Becomes self-explaining if the underline contrast is improved (P2 above).

**Fix:** Either add a `<Tooltip>` for Repo with terse copy ("Registered repository name"), or leave plain and ensure underline contrast carries the affordance.

**Suggested command:** `$impeccable polish`

## Warnings

### WR-01: CoverageToolbar debounce timer is not cleared on unmount

**File:** `packages/spa/src/components/panels/coverage/CoverageToolbar.tsx:57-98`
**Issue:** `debounceRef` holds an in-flight `setTimeout` after each keystroke,
but the component has no `useEffect` cleanup to clear it when the toolbar
unmounts. If the user types in the search field and then navigates away from
`/coverage` (or `<CoveragePage>` re-mounts because of a router transition)
within the 200 ms debounce window, the pending callback will still fire and
call `onSearchChange(value)`. That call propagates up to `<CoveragePage>`'s
`handleSearchChange`, which invokes `setSearchText` + `navigate(...)` on an
unmounted parent tree — React 18 will log a "Can't perform a React state
update on an unmounted component" warning, and in the worst case a stale
search query is written to the URL after the user has already left the page.

The Tooltip primitive (`Tooltip.tsx:34-36`) shows the correct pattern for
this codebase — an empty-deps `useEffect` whose cleanup clears the timer ref.
The same pattern needs to land here.

**Fix:**
```tsx
// CoverageToolbar.tsx — add after the existing useEffect([search])
useEffect(
  () => () => {
    if (debounceRef.current) clearTimeout(debounceRef.current)
  },
  [],
)
```

A regression test belongs in `CoverageToolbar.test.tsx` under the
`controlled input (D-11.2-13)` block:

```tsx
it('clears the debounce timer on unmount (no setState after unmount)', () => {
  vi.useFakeTimers()
  const onSearchChange = vi.fn()
  const { unmount } = render(
    <CoverageToolbar
      filter={defaultFilter}
      search=""
      onFilterChange={vi.fn()}
      onSearchChange={onSearchChange}
    />,
  )
  fireEvent.change(screen.getByRole('searchbox'), { target: { value: 'agentic' } })
  unmount()
  act(() => { vi.advanceTimersByTime(500) })
  expect(onSearchChange).not.toHaveBeenCalled()
  vi.useRealTimers()
})
```

## Info

### IN-01: Tooltip does not guard against empty `content`

**File:** `packages/spa/src/components/ui/Tooltip.tsx:24-76`
**Issue:** `content: string` is typed as required, but an empty string
(`content=""`) still renders a `<span role="tooltip">` panel that is empty.
Screen readers will read nothing on focus, which is the correct outcome, but
the empty panel adds DOM weight per column header. Callers in
`CoverageFamilySection` always pass a non-empty string from
`coverageColumnTooltips`, so this is theoretical today — but worth a guard
or a stricter type if the primitive is reused by other panels.
**Fix:** Either narrow the type with a runtime guard,

```tsx
if (!content) return <>{children}</>
```

or document the contract in the JSDoc that `content` must be a non-empty
string and rely on call-site discipline.

### IN-02: Mid-file `import type` after first use

**File:** `packages/spa/src/components/panels/coverage/CoverageFamilySection.tsx:52`
**Issue:** `import type { CoverageRowProps } from './CoverageRow.js'` is
declared on line 52, after the rest of the imports (lines 25–40) and after
the props interface that references `CoverageRowProps['onRefresh']` on
line 47. TypeScript hoists type imports, so this compiles fine, but
co-locating the type import with the value imports on line 35 keeps the
file scannable and matches the convention used elsewhere in this package.
**Fix:** Move the `import type { CoverageRowProps }` line up to sit next to
`import { CoverageRow } from './CoverageRow.js'`.

### IN-03: Inline `vi.useFakeTimers()` / `vi.useRealTimers()` without `afterEach` guard

**File:** `packages/spa/src/components/panels/coverage/CoverageToolbar.test.tsx:82-94, 224-242`
**Issue:** Two tests enable fake timers inline and reset them at the bottom of
the same test (`vi.useFakeTimers()` at the top, `vi.useRealTimers()` at the
end). If the assertion throws between the two calls, fake timers leak into
the next test in the file. The codebase's own `Tooltip.test.tsx:16-19`
demonstrates the safer pattern — restore real timers in `afterEach`
unconditionally.
**Fix:** Hoist `afterEach(() => { vi.useRealTimers() })` to the top of the
file (or to the describe blocks that flip timers). The per-test
`vi.useRealTimers()` calls can then be dropped.

### IN-04: Each column-header tooltip adds a tab stop

**File:** `packages/spa/src/components/ui/Tooltip.tsx:58-70` and
`packages/spa/src/components/panels/coverage/CoverageFamilySection.tsx:194-197`
**Issue:** The `Tooltip` trigger is a `<span tabIndex={0}>`. Wrapping each of
the four column headers (CLAUDE.md, GitNexus, Wiki, Workflow) in a tooltip
adds four extra keyboard tab stops per family section — twelve across the
three families on `/coverage`. This is the recommended WAI-ARIA pattern for
tooltips on non-interactive labels (so keyboard users can read the
description), but it materially lengthens the tab path through the page. If
UX wants to suppress the tab stop and rely purely on hover/mouse-focus
reveals, the trigger would need to drop `tabIndex={0}` and the test at
`Tooltip.test.tsx:22-32` would need to be updated.
**Fix:** None required — the current behaviour is correct per WAI-ARIA. Flag
to UX in the next IMPECCABLE pass if the tab-stop count is reported as a P2
finding; otherwise leave as-is.

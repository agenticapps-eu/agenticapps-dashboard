## Typography

**Inherited verbatim from Phase 4** (`04-UI-SPEC.md` §Typography). No new type roles in Phase 5.

| Role | Size | Weight | Line Height | Tailwind class |
|------|------|--------|-------------|----------------|
| Label | 14px (`text-sm`) | 400 regular | 1.4 | `text-sm leading-snug` |
| Body | 16px (`text-base`) | 400 regular | 1.5 | `text-base leading-relaxed` |
| Heading | 20px (`text-xl`) | 600 semibold | 1.3 | `text-xl font-semibold leading-snug` |

Phase 5-specific applications of the inherited roles:

- **Panel title** (all 5 new panels): Heading role (20px/600). Titles: `Installed Skills`, `Skill Health`, `Observability`, `Secrets`, `Integrations`.
- **Skill name** (InstalledSkills row, SkillHealth row): Label role (14px/400) but with `font-mono` (filenames + skill IDs are code-like).
- **Skill description** (InstalledSkills row, single-line truncated): Label role (14px/400) in `text-[--text-muted]`. Use `truncate` to clamp at one line. Full description NOT shown (Claude Code's 1,536-character description cap [05-RESEARCH.md §Pattern 2 footnote] makes inline expansion noisy).
- **Score badge** (SkillHealth row): `font-mono text-sm tabular-nums text-[--text]`. Format: `{N}/100`. Examples: `87/100`, `100/100`.
- **Findings count badge** (SkillHealth row): `font-mono text-xs tabular-nums text-[--text-muted]`. Format: `{N} findings`. Examples: `3 findings`, `0 findings`.
- **Severity glyph** (SkillHealth findings expansion): Unicode emoji (⚪ / 🟠 / 🔴), `aria-hidden="true"`, paired with `aria-label` on the parent row. Severity carried through `aria-label` text (e.g. `aria-label="warning: missing description field"`).
- **AgentLinter rule name** (SkillHealth expanded row): Label role (14px) in `font-mono text-[--text-muted]`. Example: `[position-risk]`, `[missing-description]`.
- **Provenance copy** (ObservabilityHealth signal row): Label role (14px/400), `text-[--text-muted]` for the leading "detected via" prefix and `text-[--text]` for the signal list. Example: `detected via @sentry/node + .sentryclirc`.
- **State badge** (SecretsHealth, IntegrationsHealth): `text-xs uppercase tracking-wide rounded px-1.5 py-0.5`. Three pills declared in §Color below.
- **Configure-to-enable copy** (IntegrationsHealth `not-detected` state): Body role (16px/400) in `text-[--text-muted]`. One paragraph per integration (D-5-20). 50–60 words per spec line 504.
- **Install-hint command** (SkillHealth `not-installed` state, IntegrationsHealth `present-but-not-configured` state): rendered through the existing `<CodeBlock>` component (Phase 3 reuse, same as Phase 4 D-4-15). `font-mono text-sm` — handled inside `CodeBlock`.

```
AppShell (sticky h-14 header + CommandPalette + RepairBanner)

        └── grid  [grid-cols-[1fr_1.5fr_1fr] gap-6 items-start]   ← changed from `1fr_1.5fr`
            ├── LeftColumn    (Discipline — Phase 4 panels, unchanged)
            ├── CenterColumn  (Phase Progress — Phase 4 panels, unchanged)
            └── RightColumn   (Health — NEW: 5 panels, Phase 5)
```

**Single-line CSS change:** `data-testid="single-project-grid"` className `grid-cols-[1fr_1.5fr]` → `grid-cols-[1fr_1.5fr_1fr]`. No other layout work.

**New section element:**

```html
<section
  data-testid="health-column"
  aria-label="Health"
  class="flex flex-col gap-4">
  <SkillHealth projectId={projectId} />
  <ObservabilityHealth projectId={projectId} />
  <SecretsHealth projectId={projectId} />
  <IntegrationsHealth projectId={projectId} />
  <InstalledSkills projectId={projectId} />
</section>
```

**Panel order (top-to-bottom in the right column):** SkillHealth → ObservabilityHealth → SecretsHealth → IntegrationsHealth → InstalledSkills. Rationale (revised during D-5-10 closure UAT — original order put the long reference list on top and crowded out the actionables):

1. **SkillHealth first** — AgentLinter surfaces actionable warnings/errors. Highest priority.
2. **Observability / Secrets / Integrations next** — three configuration-state panels. Each row is a status indicator ("configured" / "unconfigured" / "broken") with high signal density and short height. They read as a coherent triad.
3. **InstalledSkills last** — a long reference list of every globally installed skill (often 100+ entries). Belongs at the bottom of the column so it doesn't push the actionable panels off-screen on a typical 1080–1200px viewport.

**No responsive break-out for Phase 5.** D-5-01 is explicit: "no responsive break-out for Phase 5; narrow-screen stacking is for the planner to size if it shows up in QA." Below ~1024px the 3-column grid will render with the third column squeezed; the planner judges whether to ship a `lg:` breakpoint that stacks columns vertically. **Recommendation:** add `@media (max-width: 1023px)` Tailwind class `lg:grid-cols-[1fr_1.5fr_1fr] grid-cols-1` so the third column stacks below the center on narrow screens; if planner agrees, add `lg:` prefix to all grid utilities. **Defer the final call to the planner per D-5-01.**

### Panel container — Phase 4 reuse

All 5 Phase 5 panels mount the existing `<PanelContainer panelId="..." title="..." [unreachable]>` component. **No new container component.** The error / loading / unreachable / schema-drift state contracts from Phase 4 (`04-UI-SPEC.md` §State Contracts) apply unchanged:

- `isLoading || !data` → `<p class="text-sm text-[--text-muted]">Loading...</p>` body.
- `error.message.startsWith('schema_drift:')` → `<InlineDrift panelId={...} title={...} path={...} onRetry={...} />`.
- Other error → `<PanelContainer ... unreachable>{null}</PanelContainer>` (renders the inline `Agent unreachable — retrying...` row).

### Always-expanded discipline (D-4-13 carries) — and the row-level interaction exception

Panels in Phase 5 are **always-expanded** (no panel-level disclosure, no max-height clamp, no hover-expand) — Phase 4 D-4-13 carries forward unchanged.

**One exception, scoped to row-level inside SkillHealth (D-5-16):** clicking a skill row toggles inline expansion of that row to show AgentLinter findings + rule names. This is **row-level content navigation, not panel-level disclosure** — the panel itself stays fully visible at all times; only the row's detail block toggles. Other rows render normally; expansion is independent per row. Anti-slop: no animation on expand/collapse (Phase 3 D-43 carries — `display: none` ↔ `display: block` is acceptable; height transitions are not).

## Panel-level Interaction Contracts

### InstalledSkills (HEALTH-01)

**Data sources:**
- `GET /api/skills/global` → `GlobalSkillsResponseSchema` (singleton, no projectId per D-5-12)
- `GET /api/projects/:id/skills/local` → `LocalSkillsResponseSchema`

The SPA calls both queries in parallel via `useGlobalSkills()` + `useLocalSkills(projectId)`, merges the results client-side, tags each entry with `scope: 'global' | 'local'`, and renders one row per skill.

**Happy path:** Vertical list of skill rows. Each row:

```
[scope-pill]  [skill-name]                          [description, single-line truncated]
```

- `scope-pill`: `<span class="inline-flex items-center rounded bg-[--surface-elevated] px-1.5 py-0.5 text-xs uppercase tracking-wide text-[--text-muted]">global|local</span>`.
- `skill-name`: `font-mono text-sm text-[--text]` (skill ID, kebab-case).
- `description`: `text-sm text-[--text-muted] truncate flex-1` — single-line truncation with `text-overflow: ellipsis`. Full description does NOT expand inline (avoids competing with the existing SkillHealth row-level expansion).
- Row separator: `divide-y divide-[--border]`.
- Row order: globals first (alphabetical), then locals (alphabetical). Stable sort across renders.

**Empty states:**

| State | Condition | Copy |
|-------|-----------|------|
| No global + no local skills | both queries return `entries: []` | `No skills installed. Install with \`claude skill install <name>\` or place a SKILL.md under \`~/.claude/skills/\`.` |
| Global empty, local present | `global.entries === [] && local.entries.length > 0` | Render local rows only; no separator placeholder |
| Local empty, global present | `local.entries === [] && global.entries.length > 0` | Render global rows only; no separator placeholder |

**Multi-line description handling:** Phase 5 renders only the first non-empty line of `description`. If the SKILL.md uses YAML `description: |` block-scalar, the daemon-side parser collapses it to a single line for the panel response. Full multi-line text is NOT delivered to the SPA (saves bandwidth + matches the panel's row layout).

### SkillHealth (HEALTH-02) — D-5-15, D-5-16

**Data source:** `GET /api/projects/:id/agentlinter` → `AgentLinterResponseSchema` with discriminated `kind: 'ok' | 'not-installed' | 'timeout' | 'error' | 'unparseable'` (per 05-VALIDATION.md task `agentlinter/runner`).

**Happy path (kind: ok):** Vertical list of skill rows. Each row collapsed by default:

```
[skill-name]   [score-badge]   [findings-count]   [severity-glyph-row]
```

- `skill-name`: `font-mono text-sm text-[--text] flex-1`.
- `score-badge`: `font-mono text-sm tabular-nums` — color from §Color severity table (`≥ 90` success, `60–89` text, `< 60` warning). Format `{N}/100`.
- `findings-count`: `font-mono text-xs tabular-nums text-[--text-muted]`. Format `{N} findings`. `0 findings` is rendered explicitly (positive signal — matches Phase 4 D-4-14 RationalizationFires `0 fires` rule).
- `severity-glyph-row`: inline-flex row of glyphs per severity present, e.g. `🔴 1 ⚪ 2`. Glyphs `aria-hidden="true"`; row carries `aria-label="1 error, 2 info"`. If no findings: this row is omitted entirely (no `0` placeholder).
- Row container: `<button type="button">` so the click toggles the inline expansion. `aria-expanded` flips with state.

**Row expansion (D-5-16):** Clicking a row reveals an inline detail block below the same row, showing each finding:

```
  ⚪ [missing-description]  Skill description should be at least 20 characters
  🟠 [position-risk]        Skill referenced after position guard …
```

- Container: `mt-2 rounded bg-[--surface-elevated] p-3 flex flex-col gap-2`.
- Each finding row: `flex items-baseline gap-2 text-sm`.
- Glyph: `aria-hidden="true"` Unicode (⚪ / 🟠 / 🔴).
- Rule name: `font-mono text-xs text-[--text-muted]`, bracketed (e.g. `[position-risk]`).
- Diagnostic message: `text-sm text-[--text]`.

**Severity glyph mapping (resolves CONTEXT D-5-16 + RESEARCH §Open Question 1):**

| AgentLinter severity (3 emitted) | Glyph | Phase 4 D-4-16 four-glyph parity |
|----------------------------------|-------|----------------------------------|
| `error` | 🔴 | maps to **critical** bucket |
| `warning` | 🟠 | maps to **high** bucket |
| `info` | ⚪ | maps to **low** bucket |
| (no AgentLinter signal) | 🟡 medium | **unused in this panel** — see "Glyph asymmetry note" below |

**Glyph asymmetry note (open question for the planner / UI checker):** AgentLinter emits exactly three severities in 2026 (`info` / `warning` / `error`) per 05-RESEARCH.md §Summary item 1. Phase 4 D-4-16's four-glyph palette (🔴/🟠/🟡/⚪) was authored for the ReviewStatus bucket where four severities exist. **SkillHealth uses 3 of the 4 glyphs by design.** Phase 5 does not collapse 🟠 and 🟡 into a single glyph — that would lie about Phase 4 ReviewStatus's actual severity vocabulary. **Recommended resolution:** keep the 3-of-4 mapping above; the missing 🟡 is an honest reflection of AgentLinter's vocabulary, not a UI gap. The UI checker should accept this asymmetry; the planner should NOT introduce a 4th synthetic AgentLinter signal source to fill 🟡.

**Empty states (per D-5-15 — four explicitly designed states):**

| Failure class | Daemon `kind` | Copy + interaction |
|---------------|---------------|--------------------|
| AgentLinter not installed / network unreachable | `not-installed` | Body: `AgentLinter not installed. Install with \`npm install -g agentlinter\` to enable scoring.` Render via `<CodeBlock command="npm install -g agentlinter" copyLabel="Copy install command" />`. |
| Scan timeout (>30s) | `timeout` | Body: `Lint scan timed out after 30 seconds.` Followed by a `<button>` labeled `Retry scan` (44×44 touch target, `border border-[--border] rounded px-3 py-2 text-sm focus-visible:ring-2 focus-visible:ring-[--ring]`). Clicking calls `query.refetch()` with the `bypassCache: true` query parameter (D-5-15: bypasses 1h cache for one call). |
| Non-zero exit, parseable stderr | `error` | Body: `Lint scan failed.` followed by the linter's `stderr` rendered in a `<pre class="whitespace-pre-wrap rounded bg-[--surface-elevated] p-3 font-mono text-xs text-[--text-muted]">`. Exit code in trailing label: `<p class="mt-2 font-mono text-xs text-[--text-subtle]">Exit code: {N}</p>`. |
| Non-zero exit, unparseable | `unparseable` | Body: `Lint scan failed (exit {N}) — see daemon log.` No expansion, no retry. |

**Interaction contract:**
- Row click toggles `aria-expanded` and the detail block.
- Keyboard: row is focusable (`tabIndex` natural via `<button>`). Enter/Space toggles. Esc on an expanded row collapses.
- Multiple rows can be expanded simultaneously — no accordion-style "only one open" rule.
- No animation on expand/collapse (Phase 3 D-43 carries; D-4-13 carries).
- Rendering order: skills sorted by ascending score (worst first — calls user attention to the action target). Tied scores sorted alphabetically by skill name.

**Cached-stale-fallback explicitly NOT used (D-5-15):** "A stale lint scan against an outdated skill set lies more than a clear failure state." If the cached result is older than 1h AND a fresh scan fails, the panel renders the failure state, NOT the stale cached data.

### ObservabilityHealth (HEALTH-03) — D-5-17

**Data source:** `GET /api/projects/:id/observability` → `ObservabilityResponseSchema` returning a per-tool signal map.

**Happy path:** Three tool sub-sections (Sentry, Spotlight, sentry-cli), each with a one-line provenance string showing which signals matched:

```
Sentry        detected via @sentry/node + .sentryclirc
Spotlight     not detected
sentry-cli    detected via .github/workflows/release.yml
```

- Tool name: `text-sm font-semibold text-[--text]` left-aligned in a 2-column row layout (`grid grid-cols-[8rem_1fr] gap-3`).
- Provenance copy: `text-sm text-[--text]` for the matched signal list, prefixed with `text-[--text-muted]` `detected via ` text. Multi-signal joined with ` + ` (e.g. `@sentry/node + .sentryclirc`).
- "not detected" is rendered in `text-sm text-[--text-muted] italic`.

**Signal vocabulary (per D-5-17 + 05-RESEARCH.md §Summary item 6):**

| Tool | Signals (ANY-OR) |
|------|------------------|
| Sentry | `package.json` deps include `@sentry/*` ; `package.json` scripts mention `sentry-cli` ; `<root>/.sentryclirc` exists ; env file mentions `SENTRY_DSN` |
| Spotlight | `package.json` deps include `@spotlightjs/*` ; `<root>/.spotlight/` directory present |
| sentry-cli | `which sentry-cli` exit 0 ; `package.json` scripts mention `sentry-cli` ; CI YAML grep match in `.github/workflows/*.yml` |

**Empty state (no signals across any tool):**

> No observability tooling detected. (Configure to enable.)

Single body line in `text-base leading-relaxed text-[--text-muted]`. The `(Configure to enable.)` parenthetical mirrors spec line 504 verbatim phrasing without re-routing to IntegrationsHealth (different panel, different data — this empty state is honest, not redirect-y).

### SecretsHealth (HEALTH-04) — D-5-18

**Data source:** `GET /api/projects/:id/secrets` → `SecretsResponseSchema` with discriminated `state: 'present-valid' | 'present-invalid' | 'absent'`.

**Happy path:** Single-row panel — secrets state is binary-ish (3 states), so no list. The panel body renders one of three content blocks:

| State | Render |
|-------|--------|
| `present-valid` | `CheckCircle2` (16px) in `text-[--success]` + `<span class="text-sm text-[--text]">.infisical.json present and valid.</span>` + state pill `valid` |
| `present-invalid` | `AlertTriangle` (16px) in `text-[--danger]` + `<span class="text-sm text-[--text]">.infisical.json found but not parseable.</span>` + state pill `invalid` |
| `absent` | `Minus` (16px) in `text-[--text-subtle]` + `<span class="text-sm text-[--text-muted]">No .infisical.json detected.</span>` (no state pill — absent isn't a state worth pillarizing) |

- Filename rendered as `font-mono` text inline within the sentence (e.g. `<code class="font-mono">.infisical.json</code>`).
- No state pill renders for `absent` — empty state stays quiet by design.
- No interaction: SecretsHealth is purely informational. No links, no buttons, no copy actions.
- **No content from the file is ever rendered** (D-5-18 + 05-VALIDATION.md `T-5-NoSecretRead`). The panel surfaces *file presence + validity*, not workspace ID, environment names, or any secret content.

**Empty state:** the `absent` render IS the empty state. No additional copy.

### IntegrationsHealth (HEALTH-05) — D-5-19, D-5-20

**Data source:** `GET /api/projects/:id/integrations` → `IntegrationsResponseSchema` returning per-integration `state: 'configured' | 'present-but-not-configured' | 'not-detected'` for Sentry, Linear, Infisical.

**Happy path:** Three sub-sections (Sentry, Linear, Infisical), each rendering a single state row + (in `not-detected` only) inline configure-to-enable paragraph.

```
Sentry      [configured]
Linear      [set up needed]   Sentry SDK detected. Set LINEAR_API_KEY to enable the panel.
Infisical   [not detected]
            Infisical loads secrets from a Universal Auth project at runtime. Pass
            `infisical run --env=prod -- agentic-dashboard start` and the daemon
            picks up your env without code change. No env var to set inside the
            dashboard.
```

- Integration name: `text-sm font-semibold text-[--text]` in a `grid grid-cols-[7rem_1fr] gap-3` layout.
- State pill: declared in §Color above. Pill text matches:
  - `configured` (success-colored)
  - `set up needed` (warning-colored) — explicitly not `partial`. "Set up" is a clearer instruction than "partial" or "incomplete".
  - `not detected` (muted)
- `present-but-not-configured` row: pill + one-line nudge with the env var name in `font-mono`. Example: `Sentry SDK detected. Set \`SENTRY_AUTH_TOKEN\` to enable the panel.` (D-5-19 verbatim).
- `not-detected` row: pill + the inline one-paragraph "Configure to enable" guide (D-5-20). Three guides, one per integration; copy declared in §Copywriting Contract below. Each guide MUST include: install command (in `font-mono`), env var name (in `font-mono`), and what gets enabled when configured.

**Critical inheritance from D-5-20:** No external doc page link. No "Read more". No modal. The one-paragraph guide IS the documentation. Phase 6 polish may tighten copy if impeccable critique flags it; Phase 5 ships this copy verbatim.

**Empty state:** when all three integrations are `not-detected`, the panel renders three identical-shaped rows with their respective guides. The panel does NOT collapse to a single "nothing configured" empty state — three distinct guides per spec line 504 ("a single one-paragraph guide" PER integration).

## State Contracts (Phase 5 panels)

### Stale data / freshness — inherited from Phase 4

All Phase 5 queries follow the Phase 4 cadence (5s polling for short-TTL routes; 60s for `/api/skills/global`; 1h+mtime for `/api/projects/:id/agentlinter`). The 15-second per-panel stale threshold (`04-UI-SPEC.md` §Stale data) carries forward unchanged for the 5s-cadence queries.

**TTL exceptions specific to Phase 5:**

| Panel / query | Polling cadence | Stale threshold (per panel) |
|---------------|-----------------|------------------------------|
| `useGlobalSkills()` | 60s `staleTime` (planner default per CONTEXT discretion) | 180s (3× cadence) before "stale" pill appears |
| `useLocalSkills(id)` | 60s | 180s |
| `useAgentLinter(id)` | 1h hard ceiling (mtime-keyed) | NO stale pill — staleness is intrinsic to the cache contract; the panel renders cached data freely until the 1h elapses |
| `useObservability(id)` | 5s | 15s |
| `useSecrets(id)` | 5s | 15s |
| `useIntegrations(id)` | 5s | 15s |

**`useAgentLinter` exception explained:** the 1h cache is an explicit design contract (D-5-14 — "1h hard ceiling on top of mtime invalidation"). Showing a "stale" pill on cached AgentLinter data would imply something is wrong; the cache is correct by design until SKILL.md mtimes change. Anti-pattern: do not surface stale-data warnings on intentionally-cached data.

- Initial loading → `Loading...` text in `text-sm text-[--text-muted]`. No skeleton shimmer (Phase 3 D-42 carries).

## Accessibility Contract

Phase 5 inherits Phase 4's accessibility rules (`04-UI-SPEC.md` §Accessibility Contract) and adds the row-level interaction rules for SkillHealth:

- Every `<section>` panel has `aria-labelledby` pointing to its `<h2>` panel title (Phase 4 rule, applies to all 5 new panels).
- All Unicode glyph severity indicators (🔴/🟠/⚪) are `aria-hidden="true"`. Severity meaning conveyed via the parent row's `aria-label` text (e.g. `aria-label="warning: missing description field"`) or the visible body text alongside the glyph.
- **SkillHealth row buttons:**
  - Implemented as `<button type="button" aria-expanded={isExpanded}>`.
  - `aria-controls={detailRegionId}` references the expanded panel's `id`.
  - `aria-label` includes summary state ("expand details" / "collapse details") to give screen-reader context.
  - Expanded detail region: `role="region" aria-labelledby={rowId}`.
  - Esc key handler scoped to the row's expanded region collapses + returns focus to the row button.
- **SkillHealth retry button (timeout state):** `<button type="button" aria-label="Retry agentlinter scan, bypassing cache">` — explicit because the visible text "Retry scan" omits the cache-bypass detail that matters to assistive tech users.
- **Pills (state badges):** rendered as `<span>`. NOT focusable. Their meaning is also conveyed via adjacent text (e.g. SecretsHealth body sentence; IntegrationsHealth nudge or paragraph) — color is not the sole channel.
- **`<CodeBlock>` reuse** (SkillHealth `not-installed`): existing `role="status"` sr-only live region for "Copied" feedback applies unchanged.
- **Color contrast:** All severity-colored text (`--success`, `--warning`, `--danger`, `--text-muted`) on `--surface` / `--surface-elevated` MUST meet WCAG AA 4.5:1 (verified for Phase 4 against the same tokens; Phase 5 reuses them).
- **Focus rings:** `focus-visible:ring-2 focus-visible:ring-[--ring]` on every interactive element (SkillHealth row buttons, retry button, `CodeBlock` copy buttons). Inherits Phase 2.
- **Tab order in IntegrationsHealth:** No interactive elements inside the panel. The panel content is read in DOM order by screen readers; no tab stops introduced.
- **`<time>` element usage:** SkillHealth does NOT render timestamps inline. ObservabilityHealth does NOT render timestamps inline. SecretsHealth does NOT render timestamps inline. IntegrationsHealth does NOT render timestamps inline. (Timestamps belong on Phase 4 panels.)

## Open Questions (for the planner / UI checker to flag)

1. **3-col responsive break-out at narrow widths** — D-5-01 explicitly defers this to the planner. Recommendation in §Layout Contract is `grid-cols-1 lg:grid-cols-[1fr_1.5fr_1fr]` so columns stack below 1024px. Planner picks; UI checker should accept either decision but flag any width below ~960px where the third column squeezes the first two without stacking.
2. **Panel order in the right column** — Recommended top-to-bottom: InstalledSkills → SkillHealth → ObservabilityHealth → SecretsHealth → IntegrationsHealth. Rationale in §Layout Contract. Planner may reorder; UI checker validates the final order against information density (most-action-needed at top).
3. **Severity glyph asymmetry (3 used of 4)** — SkillHealth maps AgentLinter's 3 severities (`info` / `warning` / `error`) to ⚪ / 🟠 / 🔴; 🟡 medium remains unused in this panel by design. **Recommended resolution: keep the 3-of-4 mapping as-is.** Do NOT collapse 🟠 + 🟡 — that would lie about Phase 4 ReviewStatus's 4-bucket reality. Do NOT introduce a synthetic 4th AgentLinter signal source. UI checker should accept this asymmetry; the Phase 4 ReviewStatus panel continues to use all 4 glyphs.
4. **"Position Risk" naming alignment** — Both 05-CONTEXT.md (D-5-16) and the spec use the phrase "Position Risk" but AgentLinter's actual JSON output uses generic `severity` + `category` fields per 05-RESEARCH.md §Summary. **Recommended resolution:** rename the panel content from "Position Risk warnings" to "AgentLinter findings" (or just "findings") in the rendered copy. Phase 5 UI-SPEC's row-format string `{N} findings` already reflects this. The phrase "Position Risk" appears nowhere in user-visible copy; it remains as a CONTEXT.md / spec term-of-art only.
5. **`useAgentLinter` cache-bypass UX on retry** — D-5-15 says the `Retry scan` button bypasses the 1h cache for one call. The implementation contract is in 05-VALIDATION.md `agentlinter/route` task. UI-SPEC declares the button label and behavior; planner declares the query parameter shape (e.g. `?bypassCache=1`).

Create `.github/workflows/release.yml` following RESEARCH §Pattern 8 verbatim. Use the same action pin versions as `ci.yml` (`actions/checkout@v6`, `pnpm/action-setup@v6`, `actions/setup-node@v4`).

```yaml
name: Release

on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    name: Publish @agenticapps/dashboard-agent
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write   # Required for npm provenance attestation

    steps:
      - name: Checkout repository
        uses: actions/checkout@v6

      - name: Setup pnpm
        uses: pnpm/action-setup@v6
        with:
          version: 10

      - name: Setup Node.js (with npm registry auth)
        uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'
          cache: 'pnpm'
          registry-url: 'https://registry.npmjs.org'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Lint
        run: pnpm lint

      - name: Type check
        run: pnpm typecheck

      - name: Test
        run: pnpm test --run

      - name: Build agent
        run: pnpm --filter @agenticapps/dashboard-agent build

      - name: Validate package exports (publint)
        working-directory: packages/agent
        run: pnpm dlx publint

      - name: Validate type exports (arethetypeswrong)
        working-directory: packages/agent
        run: pnpm dlx --package=@arethetypeswrong/cli attw --pack .

      - name: Publish to npm with provenance
        working-directory: packages/agent
        run: pnpm publish --provenance --access public --no-git-checks
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

KEY POINTS (RESEARCH §Pattern 8 + §Pitfall 3 — DO NOT REMOVE):
- `permissions: id-token: write` is REQUIRED for npm provenance. Without it, `--provenance` fails at the publish step.
- `registry-url: 'https://registry.npmjs.org'` on `setup-node` creates the `.npmrc` file so `NODE_AUTH_TOKEN` is honored. Without it, `npm publish` returns 401.
- `--no-git-checks` because we publish from a tag checkout (clean working tree but pnpm sometimes complains about non-default branch).
- `--access public` REQUIRED on first publish of a scoped package. After the first publish it becomes optional but harmless.
- `working-directory: packages/agent` ensures `pnpm publish` targets the right package. Alternative `pnpm --filter @agenticapps/dashboard-agent publish` from repo root also works.
- We run `lint`, `typecheck`, `test`, `build` BEFORE publish (D-12 — same gates as ci.yml). Repeating them in the release workflow protects against tag-push from a stale branch.
- `publint` validates `package.json` exports + bin shape; `attw` validates `.d.ts` types match the JS export structure.

Pitfall avoidance:
- DO NOT add Cloudflare deploy steps to release.yml (D-11 — CF Pages Git integration handles deploys).
- DO NOT add changesets logic (D-08 — manual version bumps in Phase 0).
- DO NOT skip the lint/typecheck/test gates (D-12).

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Tag push → GH Actions runner | Tag presence is the gate; any maintainer with push access can trigger a publish |
| Runner → npm registry | OIDC token + provenance attestation; `NPM_TOKEN` is the auth surface |
| npm → consumers | Published artifact + provenance statement; `--access public` makes it world-readable |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-04 | T (Tampering) | Published agent bundle | mitigate | (a) `--provenance` adds SLSA Level 3 attestation linking the artifact to the GH Actions run; (b) `publint` + `attw` catch malformed exports; (c) lint/typecheck/test gates run on the tag commit before publish; (d) `id-token: write` permission scoped to release workflow only. |
| T-00-11 | E (Elevation of Privilege) | `NPM_TOKEN` secret | mitigate | (a) Token never exposed in workflow logs (GH Actions auto-redacts secrets); (b) `permissions:` block restricts `contents: read` so workflow cannot push commits/tags back to the repo; (c) only `release.yml` accesses `NPM_TOKEN` — `ci.yml` never references it. Future hardening: rotate to short-lived OIDC publish token when npm registry supports it directly. |
| T-00-12 | S (Spoofing) | Tag push trigger | accept | Anyone with push access to the repo can push a `v*` tag and trigger publish. Repo is private with single user (D-13). Re-evaluate at Phase 8 public flip — at that point, gate via Tag Protection Rules in GH settings. |

</threat_model>

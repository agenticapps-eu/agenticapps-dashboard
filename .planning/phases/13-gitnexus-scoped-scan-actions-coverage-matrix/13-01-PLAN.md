<objective>
Plumb `bindMode` through the daemon's Hono `Env.Variables` so Wave 2's new `/api/gitnexus/scan` route can read it via `c.get('bindMode')` to enforce the D-13-11 loopback-only refusal (Pitfall 2 in 13-RESEARCH.md). Two changes, narrow scope:

1. **`bindMode` plumbing.** Currently `BootOptions.bindMode` lives in `boot.ts` (`'loopback' | 'tailscale' | '0.0.0.0'`) but is NOT exposed to route handlers. Phase 13 must:
   - Add `bindMode` to `app.ts:CreateAppOptions`
   - Add `bindMode` to `app.ts:Env.Variables` type
   - Set it in the middleware closure (next to `registryFile` / `authFile` pattern at ~line 80)
   - Thread through `boot.ts → createApp()` and `cli/start.ts → boot()`

2. **Health response extension.** `/health` returns a new `gitnexus: { installed, canScan }` object:
   - `installed` = result of `detectGitNexusBinary()` (Phase 10.6's stat-based probe — survives launchd minimal PATH per Phase 10.6 D-10.6-02)
   - `canScan` = `installed && bindMode === 'loopback'` (D-13-11b — composite the SPA reads to render Scan pills)
   - Wire schema in `@agenticapps/dashboard-shared` extended with `.strict()` discipline

This wave is parallel-safe from Wave 0 because the only schema dependency is `HealthResponseSchema` in shared, which already exists. Plan 13-02 will consume `c.get('bindMode')` in the new scan route.
Purpose: pre-establish the runtime signal Wave 2's bind-refusal logic + Wave 3's SPA tooltip both depend on.
Output: `bindMode` flowing through every Hono request; `/health` response carrying `gitnexus: { installed, canScan }`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser SPA → daemon `/health` | The health response is the SPA's signal for whether to render Scan affordances enabled (D-13-11b). Bearer-auth required (inherited from `app.use(bearerAuth(...))`). |
| `bindMode` runtime signal → `/health` handler | bindMode flows from CLI flag → BootOptions → createApp → Env.Variables → route handler. Each hop is in-process; no cross-trust-boundary transmission. |
| Daemon → `~/.gitnexus/` filesystem stat (via detectGitNexusBinary) | `detectGitNexusBinary` performs read-only filesystem stat probes against well-known binary install prefixes (Phase 10.6 — XDG, fnm, nvm, npm-global, volta, bun, homebrew, /usr/local). Read-only. No write boundary touched. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01-01 | Tampering | `Env.Variables.bindMode` could be set to an unsafe default if CreateAppOptions.bindMode is missing | mitigate | Explicit `opts.bindMode ?? 'loopback'` default — the SAFEST mode (refuses scan routes). If CLI parsing fails silently, the daemon still refuses remote scan attempts. Loopback is the principle-of-least-privilege default. |
| T-13-01-02 | Information Disclosure | `/health` response leaking system state (binary install paths, env vars) | mitigate | `gitnexus.installed` is a boolean only — does NOT include the resolved path, version, or any other detail. `canScan` is a boolean. No PII or system-internal data exposed. |
| T-13-01-03 | Elevation of Privilege | Route handler trusting an unauthenticated bindMode signal | mitigate | bindMode is set by the daemon's own startup code in `cli/start.ts` (CLI flag parsing). It is NEVER set from HTTP request headers or body. The Variables value is established server-side at app creation time, immutable for the app's lifetime per request. |
| T-13-01-04 | Spoofing | Attacker spoofs `/health` to claim canScan=true when the daemon is actually on tailscale | accept (no mitigation needed) | The SPA reads `/health` from the same daemon it intends to scan. If the daemon is compromised, the threat surface is the entire daemon, not this field. Bearer-auth + CORS lock prevent cross-origin spoofing. |
| T-13-01-05 | Tampering (carve-out documented; for full mitigation see Plan 13-02) | `detectGitNexusBinary` performs stat against $HOME-resolved paths | accept | Phase 10.6 already shipped this code. Stat-based; no shell-out; no symlink-target-read. Worst case: a malicious symlink at `~/.local/bin/gitnexus` resolves to a non-gitnexus binary. Wave 2 mitigates the spawn-level risk via `which`-based resolution + execa argv-array form. This Wave's `installed: boolean` just reflects whether SOME path exists — not whether it's legitimate. |
</threat_model>
